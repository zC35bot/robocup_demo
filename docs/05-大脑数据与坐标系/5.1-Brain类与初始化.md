# 5.1 · Brain 类与初始化

`Brain` 是整个大脑进程的总容器。本篇讲它的类结构、构造函数里那一大串参数声明、以及 `init()` 如何把所有子系统装配起来。

---

## 一、类结构（`brain.h:66`）

`Brain` 继承自 `rclcpp::Node`（所以它本身就是一个 ROS2 节点），并把功能拆进若干**子对象**：

```cpp
class Brain : public rclcpp::Node {
public:
    std::shared_ptr<BrainConfig> config;          // 静态配置（只读）→ 5.2
    std::shared_ptr<BrainLog>    log;             // 日志
    std::shared_ptr<BrainData>   data;            // 动态数据（实时）→ 5.3
    std::shared_ptr<RobotClient> client;          // 机器人控制接口 → 模块08
    std::shared_ptr<Locator>     locator;         // 自定位 → 模块06
    std::shared_ptr<VisualizationPublisher> visualizer;  // 可视化 → 模块08
    std::shared_ptr<BrainTree>   tree;            // 行为树 → 模块07
    std::shared_ptr<BrainCommunication> communication;   // 队内通信 → 模块04
    ...
private:
    // Phase1 组件（直接成员，非指针）
    BallImmPredictor      imm_predictor_;         // 场地系球预测 → 模块06
    RobotFramePredictor   robot_frame_predictor_; // 机器人系球预测 → 模块06
    HeadController        head_controller_;        // 主动头控 → 模块08
    TrainingLogger        training_logger_;        // 训练数据记录 → 模块08
    std::mutex            predictorMutex_;         // 保护预测器共享状态
};
```

> 💡 `brain.h:60` 的注释明确写了设计纪律：**"数据封装进子对象，而不是直接堆在 Brain 里。静态配置放 config，动态运行数据放 data。"** 行为树节点会拿到 `brain` 指针，直接访问 `brain->config`、`brain->data`、`brain->client` 等——所以这些子对象用 `shared_ptr`，能安全地被多处共享。`Brain` 还要支持 `shared_from_this`，因为它的指针被传进行为树。

类里还声明了**一大批订阅句柄和发布句柄**（`brain.h` 后半部分的 private 成员），以及所有回调函数和辅助函数的声明。这些回调的实现散落在 `brain.cpp`，分别在 [模块06](../06-定位与球预测/index.md)（感知）、[模块07](../07-行为树与决策/index.md)（决策辅助）、[模块08](../08-机器人控制与底层/index.md)（控制）讲。

## 二、构造函数：声明所有参数（`brain.cpp:19`）

```cpp
Brain::Brain() : rclcpp::Node("brain_node") {
    tf_broadcaster_ = std::make_shared<tf2_ros::TransformBroadcaster>(*this, rclcpp::QoS(10).transient_local());
    declare_parameter<int>("game.team_id", 0);
    declare_parameter<bool>(GAME_AGENT_MODE_PARAM, false);
    ...
}
```

构造函数干一件事：**用 `declare_parameter` 声明它将要读取的所有 ROS 参数及默认值**。这是 ROS2 的硬性要求——参数必须先声明才能读。从 `brain.cpp:28` 到 `brain.cpp:172`，密密麻麻声明了上百个参数，分组对应 [模块01 · 1.3](../01-启动与架构/1.3-launch文件逐行.md) 讲的层叠配置：

```cpp
// 比赛
declare_parameter<int>("game.team_id", 0);
declare_parameter<int>("game.player_id", 1, player_id_desc);  // 默认 1，带范围校验描述符
declare_parameter<string>("game.player_role", "", player_role_desc);
// 机器人本体
declare_parameter<double>("robot.robot_height", 1.0);
declare_parameter<double>("robot.odom_factor", 1.0);
declare_parameter<double>("robot.vx_limit", 1.0);
// 策略
declare_parameter<double>("strategy.ball_confidence_threshold", 50.0);
// 避障
declare_parameter<double>("obstacle_avoidance.grid_size", 0.2);
// RLVisionKick / locator / Phase1 参数块 ...
```

> 💡 注意 `player_id` 附了一个 `ParameterDescriptor` 做**范围校验**（`brain.cpp:31`）：
> ```cpp
> // 文件：src/brain/src/brain.cpp:31
> rcl_interfaces::msg::ParameterDescriptor player_id_desc;
> player_id_desc.description = "player_id must be an integer between 1 and 11";
> player_id_desc.type = rcl_interfaces::msg::ParameterType::PARAMETER_INTEGER;
> player_id_desc.integer_range.resize(1);
> player_id_desc.integer_range[0].from_value = 1;
> player_id_desc.integer_range[0].to_value   = HL_MAX_NUM_PLAYERS;  // = 11
> player_id_desc.integer_range[0].step       = 1;
> declare_parameter<int>("game.player_id", 1, player_id_desc);      // brain.cpp:38
> ```
> 描述符把 `player_id` 限定为 **1~`HL_MAX_NUM_PLAYERS`(=11) 的整数**，并把默认值从原来的 `29` 改成合法的 `1`（旧默认值 29 本身就越界，是个隐患）。这一步很关键：后续代码大量用 `player_id - 1` 去索引 `penalty[HL_MAX_NUM_PLAYERS]`、`tmStatus[HL_MAX_NUM_PLAYERS]` 这些长度为 11 的数组（见第五节），只要 `player_id` 落在 1~11，`player_id-1` 就恒在 `[0,10]` 合法下标区间。把校验前移到"参数声明"这一层，launch 传入越界值时 ROS2 会**在启动阶段直接报错**，而不是等到运行时数组越界崩溃。
> 这里 `declare_parameter` 给的默认值，会被 launch 加载的 `config.yaml` 等覆盖（层叠机制见 [模块01](../01-启动与架构/index.md)）。声明只是"占位+兜底"。

## 三、`init()`：装配大脑（`brain.cpp:180`）

`main.cpp` 在构造后调一次 `init()`，它是真正的初始化总装。逐段：

### 1. 创建子对象 + 加载配置
```cpp
config = std::make_shared<BrainConfig>(this);
loadConfig();                               // 读 yaml、Phase1 参数（见下）
data = std::make_shared<BrainData>();
locator = std::make_shared<Locator>();
log = std::make_shared<BrainLog>(this);
tree = std::make_shared<BrainTree>(this);
client = std::make_shared<RobotClient>(this);
communication = std::make_shared<BrainCommunication>(this);
visualizer = std::make_shared<VisualizationPublisher>(this);

locator->init(config->fieldDimensions, config->get_min_marker_count(), config->get_max_residual());
tree->init();                               // 注册行为树节点、加载 XML（模块07）
client->init(config->get_robot_name());     // 建 RPC 发布句柄（模块08）
communication->initCommunication();         // 起通信线程（模块04）
```
**创建顺序有依赖**：先 `config`（别人要读配置），再 `data`/`locator`/`log`，然后 `tree`/`client`/`communication`。`locator->init` 把场地尺寸和定位参数传进去（见 [模块06](../06-定位与球预测/index.md)）。

### 2. 初始化时间戳与 Phase1 组件
```cpp
data->lastSuccessfulLocalizeTime = get_clock()->now();
data->timeLastDet = ...; data->timeLastGamecontrolMsg = ...;  // 各种"上次..."时间初始化为现在

// 配置两个球预测器（参数来自 config->ballPredictor）
BallPredictorParams pp;
pp.friction_decay_hz = config->ballPredictor.friction_decay_hz;
...
imm_predictor_.configure(pp);  imm_predictor_.reset();
robot_frame_predictor_.configure(pp);  robot_frame_predictor_.reset();

// 训练记录器
training_logger_.init(config->trainingLogger.enable, config->get_player_id(), config->get_robot_name(), startUs);

// 队友状态初始化为"全部离线"
for (int i = 0; i < HL_MAX_NUM_PLAYERS; i++) { data->tmStatus[i].isAlive = false; ... }
```

### 3. 建立话题订阅与发布（多机隔离的关键，`brain.cpp:241`）
```cpp
string topic_suffix = "";
if (config->get_robot_name() != "") {
    topic_suffix = "/" + config->get_robot_name();   // 如 "/robot0"
}
detectionsSubscription = create_subscription<...Detections>("/booster_soccer/detection" + topic_suffix, ...);
subFieldLine          = create_subscription<...LineSegments>("/booster_soccer/line_segments" + topic_suffix, ...);
odometerSubscription  = create_subscription<...Odometer>("/odometer_state" + topic_suffix, ...);
lowStateSubscription  = create_subscription<...LowState>("/low_state" + topic_suffix, ...);
headPoseSubscription  = create_subscription<...Pose>("/head_pose" + topic_suffix, ...);
recoveryStateSubscription = create_subscription<...RawBytesMsg>("fall_down_recovery_state" + topic_suffix, ...);
// 相机内参用配置里的话题名（不加后缀）
imageCameraInfoSubscription = create_subscription<...CameraInfo>(config->get_image_camera_info_topic(), ...);
depthCameraInfoSubscription = create_subscription<...CameraInfo>(config->get_depth_camera_info_topic(), ...);
```
> 💡 **`topic_suffix` 就是多机隔离的核心**（呼应 [模块01 · 1.2](../01-启动与架构/1.2-脚本家族.md)）：设了 `robot_name="robot0"`，所有业务话题就变成 `/booster_soccer/detection/robot0` 等。6 个大脑跑同一份代码，但各自只收发带自己名字的话题，互不串扰。

发布句柄也类似建立：`pubRobotPose`、`pubBallPosition`、`pubFieldDimensions`、`pubKickBall`(`/kick_ball`)、`pubTeammatesPoses` 等。

深度图订阅会根据话题名是否含 `compressed` 选择压缩/非压缩回调（`brain.cpp:270`）。

### 4. agent 模式特殊处理（`brain.cpp:280`）
```cpp
if (get_parameter(GAME_AGENT_MODE_PARAM).as_bool()) {
    // 订阅 team_id / player_role 参数变化，支持运行时热切换队伍/角色
    param_subscriber_ = std::make_shared<rclcpp::ParameterEventHandler>(this);
    team_id_handle_ = param_subscriber_->add_parameter_callback("game.team_id", team_id_cb);
    player_role_handle_ = param_subscriber_->add_parameter_callback("game.player_role", player_role_cb);
    // agent 模式没有裁判机，强制清空罚则，允许队友正常通信
    std::fill(std::begin(data->penalty), std::end(data->penalty), PENALTY_NONE);
}
```
agent 模式（脱离裁判机、由 App 控制，见 [模块04](../04-裁判机与通信/index.md)）下，`team_id` 变化时会**重建通信对象**（因为通信端口绑了 teamId），`player_role` 变化时更新行为树黑板。

最后调 `publishFieldDimensions()` 把场地尺寸发出去（供可视化）。

## 四、`loadConfig()`：读视觉外参 + Phase1 参数（`brain.cpp:324`）

`init()` 里调用，做两件 `declare_parameter` 之外的配置加载：

### 1. 读相机外参 `camToHead`
```cpp
YAML::Node vConfig = YAML::LoadFile(visionConfigPath);          // vision.yaml
if (filesystem::exists(visionConfigLocalPath)) {
    MergeYAML(vConfig, YAML::LoadFile(visionConfigLocalPath));  // 叠加 vision_local.yaml
}
auto extrin = vConfig["camera"]["extrin"];
for (int i=0;i<4;i++) for (int j=0;j<4;j++)
    config->camToHead(i,j) = extrin[i][j].as<double>();         // 4×4 外参矩阵
```
> 💡 大脑要做坐标变换（比如把深度图里的障碍点转到机器人系），需要"相机相对头部"的外参矩阵。这个矩阵存在**视觉**的 `vision.yaml` 里（见 [模块03](../03-视觉模块/index.md)），所以大脑也要读它——这就是 launch 给大脑传 `vision_config_path` 的原因（见 [模块01 · 1.3](../01-启动与架构/1.3-launch文件逐行.md)）。`MergeYAML` 实现层叠（见 [5.5](./5.5-工具函数.md)）。

### 2. 调 `config->handle()` + 读 Phase1 参数块
```cpp
config->handle();   // 选场地尺寸、算地图线/角点（见 5.2）
auto &bp = config->ballPredictor;
bp.friction_decay_hz = get_parameter("ball_predictor.friction_decay_hz").as_double();
... // 把所有 Phase1 参数从 ROS 参数读进 config 的结构体
config->headController.enable = get_parameter("head_controller.enable").as_bool();
config->trainingLogger.enable = get_parameter("training_logger.enable").as_bool();
```
最后校验 `player_role` 必须是 `striker` 或 `goal_keeper`，否则抛异常终止（`brain.cpp:404`）——**宁可启动失败也不带错配置跑**。

## 五、回调里的越界防护（内存安全加固）

除了参数声明期的范围校验，`brain.cpp` 的几个**回调**在拿网络/底层来的原始索引和缓冲区做下标访问前，都补上了显式的边界检查。这些数据来自裁判机、队友或运动控制底层，**不可信**，越界访问轻则读到脏数据、重则 `memcpy`/下标越界直接崩进程。

### 1. `penalty[]` 下标校验（`gameControlCallback`，`brain.cpp:1568`）

判断"本机是否被罚下"要用 `playerId - 1` 去索引长度为 `HL_MAX_NUM_PLAYERS`(=11) 的 `penalty[]`。虽然第二节的描述符已经把参数层的 `player_id` 夹在 1~11，这里仍做**第二重兜底**：

```cpp
// 文件：src/brain/src/brain.cpp:1568
int pidIdx = playerId - 1;
bool isUnderPenalty = (pidIdx >= 0 && pidIdx < HL_MAX_NUM_PLAYERS)
                      ? (data->penalty[pidIdx] != PENALTY_NONE)
                      : true;   // 索引非法时保守地当作"被罚下"
```
> 💡 越界时不是崩溃、也不是乐观地当"没被罚"，而是**保守地返回 `true`（当作被罚下）**——最坏情况机器人待命等待重新入场，不会因为一个坏 `playerId` 就带着越界读的脏状态继续踢。防御式编程的"故障安全"取向。

### 2. 恢复状态缓冲区大小校验（`recoveryStateCallback`，`brain.cpp:1747`）

摔倒恢复状态从底层以**裸字节 `RawBytesMsg`** 传来，回调要把字节 `memcpy` 进 `RobotRecoveryStateData` 结构体，再用其中的 `state` 枚举去查表。两处都加了校验：

```cpp
// 文件：src/brain/src/brain.cpp:1747
const std::vector<unsigned char>& buffer = msg.msg;
if (buffer.size() < sizeof(RobotRecoveryStateData)) return;   // ① 缓冲不够长，直接丢弃
RobotRecoveryStateData recoveryState;
memcpy(&recoveryState, buffer.data(), sizeof(RobotRecoveryStateData)); // 只拷结构体大小
...
int stateIdx = static_cast<int>(recoveryState.state);          // ② 枚举下标校验
if (stateIdx >= 0 && stateIdx < static_cast<int>(recoveryStateMap.size()))
    this->data->recoveryState = recoveryStateMap[stateIdx];
else
    this->data->recoveryState = RobotRecoveryState::IS_READY;  // 越界回退
```
- **① 缓冲区长度校验**：原来的写法是 `memcpy(&recoveryState, buffer.data(), buffer.size())`——若底层发来的 `buffer` 比结构体**大**，会写爆栈上的 `recoveryState`（缓冲区溢出）；若比结构体**小**，`memcpy` 又会越界读 `buffer`。改成先 `if (buffer.size() < sizeof(...)) return;` 挡掉过短包，再固定按 `sizeof(RobotRecoveryStateData)` 拷贝，两个方向的越界都堵死。
- **② 枚举下标校验**：`recoveryState.state` 是网络/底层来的整数，用它查 `recoveryStateMap`（只有 4 个元素：`IS_READY/IS_FALLING/HAS_FALLEN/IS_GETTING_UP`）前先夹区间，越界回退到最保守的 `IS_READY`。

### 3. `position_projection` 长度校验（`getGameObjects`，`brain.cpp:2025`）

视觉检测里每个物体带一个投影坐标数组 `position_projection`，代码要取它的 `[0]`(x) 和 `[1]`(y)。若视觉某帧没算出投影、数组为空或只有 1 个元素，直接取 `[1]` 就是越界：

```cpp
// 文件：src/brain/src/brain.cpp:2025
if (obj.position_projection.size() < 2) continue;   // 长度不足，跳过这个物体
gObj.posToRobot.x = obj.position_projection[0];
gObj.posToRobot.y = obj.position_projection[1];
```
不足 2 个元素就 `continue` 跳过该物体，宁可少一个观测，也不越界读一个不存在的坐标。

> 🏆 这三处加上第四节的 `player_role` 校验、第二节的 `player_id` 描述符，构成大脑"**外部数据进门先验身**"的一贯风格：凡是来自网络（裁判机/队友）、底层（运动控制）、感知（视觉）的索引和缓冲区，在下标访问或 `memcpy` 之前都必须先校验范围/长度，越界一律走**故障安全**回退分支，绝不让坏数据把进程带崩。

## 小结

- `Brain` 继承 `rclcpp::Node`，把功能拆进 `config/data/log/client/locator/tree/communication/visualizer` 等子对象。
- 构造函数只做一件事：`declare_parameter` 声明上百个参数及默认值（会被层叠配置覆盖）。
- `init()` 按依赖顺序创建子对象、初始化预测器、**用 `topic_suffix` 建立带机器人名的话题**（多机隔离）、处理 agent 模式。
- `loadConfig()` 额外读视觉外参 `camToHead` 和 Phase1 参数，并做关键校验。
- **内存安全加固**：`player_id` 参数带 1~11 范围描述符（默认值从越界的 29 改为 1）；`gameControlCallback`/`recoveryStateCallback`/`getGameObjects` 等回调在用外部索引下标访问、`memcpy` 前都先校验范围/缓冲区长度，越界走故障安全回退（见第五节）。

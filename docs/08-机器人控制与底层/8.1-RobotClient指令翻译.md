# 8.1 · RobotClient 指令翻译官

行为树的动作节点（[7.5](../07-行为树与决策/7.5-动作节点-追球调整踢球.md)）最后都调 `client->setVelocity(...)`、`client->crabWalk(...)`、`client->standUp()` 这些方法。`RobotClient`（`src/brain/src/robot_client.cpp`、`src/brain/include/robot_client.h`）就是这层**翻译官**：把"语义级意图"封成 RPC 指令发给机器人本体。本篇逐个看它的机制与每条指令。

---

## 一、底层机制：`call()`（`robot_client.cpp:16`）

所有指令最终都走 `call()`，没有例外：

```cpp
int RobotClient::call(BoosterApiReqMsg msg) {
    std::string uuid = gen_uuid();                 // 生成追踪用 uuid
    auto message = RpcReqMsg();
    message.uuid = uuid;
    nlohmann::json req_header;
    req_header["api_id"] = msg.api_id;             // api_id 放进 header
    message.header = req_header.dump();
    message.body = msg.body;                       // 参数(JSON) 放进 body
    publisher->publish(message);                   // 发到 LocoApiTopic{robot}Req
    return 0;                                       // 发了就返回，不等回执
}
```

发布话题在 `init()`（`robot_client.cpp:10`）里建：`LocoApiTopic{/robot_name}Req`（多机时按 `robot_name` 加后缀）。Robot SDK 在另一端订阅、解析 `api_id` 执行对应动作。

> 💡 **异步、不阻塞**是这层的关键设计。大脑 `publish` 完立即返回，继续跑下一帧决策，绝不等"机器人做完"。这正是 [模块02](../02-接口与消息/index.md) 讲的 RPC 消息模式（`booster_ros2_interface`）：决策层只发"以 vx 前进""起身"这种高层指令，底层 SDK 负责把它变成几十个关节的实时力矩控制。决策代码完全不用碰双足平衡这种极难的控制问题，两侧各自迭代、互不拖累。

---

## 二、指令表

`RobotClient` 暴露的高层方法都是 `call()` 的薄封装，区别只在 `api_id` 和 body：

| 方法 | 行号 | 作用 | 底层 API（`LocoApiId` / 构造器）|
|------|------|------|----------|
| `moveHead(pitch, yaw)` | 30 | 转头（追球/扫场）| `CreateRotateHeadMsg`，先按头部限位钳制 yaw/pitch，异常角度打 error 日志 |
| `setVelocity(vx,vy,vθ, applyMin…)` | 83 | 持续行走（**核心**）| `CreateMoveMsg`，含三道安全处理 |
| `crabWalk(angle, speed)` | 130 | 朝某方向蟹步（冲球用）| 内部转 `setVelocity` |
| `standUp()` | 43 | 摔倒后起身 | `kGetUp`，body 为空 |
| `RLVisionKick(start)` | 53 | 开/关强化学习视觉踢球 | `kVisualKick`，body 带 `start` 布尔与 `version=2` |
| `robocupWalk()` | 66 | 切回比赛行走步态 | 先 `RLVisionKick(false)` 退视觉踢，再 `CreateChangeModeMsg(kWalking)` |
| `enterDamping()` | 73 | 进入阻尼/软瘫（急停）| `CreateChangeModeMsg(kDamping)` |
| `waveHand(doWave)` | 78 | 挥右手开/合（进球庆祝）| `CreateWaveHandMsg(kRightHand, ...)` |
| `moveToPoseOnField/2/3` | 155/256/357 | 导航到场地某点 | 输出 `setVelocity`，详见 [8.2](./8.2-导航moveToPoseOnField.md) |

`moveHead`（`robot_client.cpp:30`）在下发前先把 `yaw` 钳进左右限位、`pitch` 不低于上限，再 `CreateRotateHeadMsg`——这是头部安全的第一道闸（[8.4](./8.4-头部主动控制HeadController.md) 的 `sendHead` 还有第二道）。钳制之后还加了一道**异常值告警**（`robot_client.cpp:35`）：

```cpp
if (fabs(pitch) > 2.0 || fabs(yaw) > 2.0)
    brain->log->error("move_head", format("ABNORMAL pitch: %.1f, yaw: %.1f", pitch, yaw));
else
    brain->log->debug("move_head", format("pitch: %.1f, yaw: %.1f", pitch, yaw));
```

> 💡 头部角度单位是弧度，正常值远小于 2.0（约 115°）。若钳制后仍出现 `|pitch|>2` 或 `|yaw|>2`，说明上游算出了离谱角度（或限位配置写错），此时把日志从 `debug` 提到 `error`，让问题在日志里立刻醒目，而不是淹没在每帧刷屏的 debug 里。

`RLVisionKick`（`robot_client.cpp:53`）的 body 里 `start=true/false` 表示进入/退出视觉踢模式，配合 [7.5](../07-行为树与决策/7.5-动作节点-追球调整踢球.md) 的 `RLVisionKick` 节点使用。它现在额外带一个 `version` 字段：

```cpp
body["start"] = start;
body["version"] = 2;   // kV2：沿用旧版视觉踢行为，固件要求带版本号
```

> 💡 固件升级后 `kVisualKick` 接口要求请求体显式携带 `version`，否则可能按默认版本或直接拒绝。显式写 `version=2` 锁定"沿用此前那套视觉踢行为"，避免固件升级后行为悄悄漂移。

`robocupWalk`（`robot_client.cpp:66`）切回比赛步态前会**先退出视觉踢模式**：

```cpp
RLVisionKick(false);   // 先关掉 VisualKick
return call(booster_interface::CreateChangeModeMsg(booster::robot::RobotMode::kWalking));
```

> 💡 之前只发 `kWalking` 切模式，但如果机器人当时正处于视觉踢状态，直接切步态可能与未退出的踢球状态机冲突。先显式 `RLVisionKick(false)` 保证从一个干净状态进入行走，避免残留状态导致的异常动作。

## 三、`setVelocity`：速度指令的三道安全处理（`robot_client.cpp:83`）

`setVelocity` 是用得最多的指令，但它**不是裸转发**，而是依次做三件事。它的签名新增了三个可选开关（`robot_client.h:47`）：

```cpp
int setVelocity(double x, double y, double theta,
                bool applyMinX = true, bool applyMinY = true, bool applyMinTheta = true);
```

> 💡 `applyMinX/Y/Theta` 分别控制对应分量**是否套用下面 3.1 的最小速度兜底**，默认全开、行为与旧版一致。之所以要能单独关掉：某些场景（如精细微调、或只想在某一轴慢速逼近目标）不希望被下限强行抬速——比如只需要一点点侧移，却被 `min_vy` 顶成一大步就会冲过头。给调用方留了按轴关闭兜底的余地，而不必改全局配置。

### 3.1 最小速度兜底（`robot_client.cpp:88`）

```cpp
double minx = config->get_min_vx(); /* miny, mintheta 同理 */
if (applyMinX && fabs(x) < minx && fabs(x) > 1e-5) x = x > 0 ? minx : -minx;   // 太小→给个下限
// y、theta 各自同样处理（分别受 applyMinY / applyMinTheta 控制）
```

> 💡 太小的速度指令机器人迈不动腿（步态有最小步长），但又不能直接当 0——那样该走的没走。所以非零但低于下限的，统一抬到下限并保留方向。注意 `> 1e-5` 的判断：真正的 0 不动它，避免"想停却被强行迈一步"。新增的 `applyMinX` 等前置条件让调用方可按轴跳过这道兜底。

### 3.2 限速钳制（`robot_client.cpp:97`）

```cpp
x     = cap(x,     config->get_vx_limit(),     -config->get_vx_limit());
y     = cap(y,     config->get_vy_limit(),     -config->get_vy_limit());
theta = cap(theta, config->get_vtheta_limit(), -config->get_vtheta_limit());
```

`cap` 把三个分量钳进 `±vx/vy/vtheta_limit`，防止决策层算出离谱速度导致跌倒/失控。这是**最后一道总闸**——无论上游怎么算，下发的速度一定在安全范围内。

### 3.3 路径仿真与记录（`robot_client.cpp:101`）

```cpp
vector<Pose2D> path = {{0, 0, 0}};
double v = norm(x, y);
double simStep = 1e-1; double simLength = 5.;          // 向前模拟 5 米
for (int i = 0; i < simLength/simStep; i++) {
    double dt = simStep * (i + 1);
    if (fabs(theta) < 1e-3) path.push_back({v*dt, 0, 0});                   // 直线
    else path.push_back({v/theta*sin(theta*dt), v/theta*(1-cos(theta*dt)), theta*dt}); // 圆弧（单车模型）
}
// 再把每个点 trans 到场地系，供可视化
```

> 💡 按当前速度用**单车运动学模型**向前推演 5 米轨迹（直行是直线、带 vθ 是圆弧），转到场地系记录下来。这样在 Booster Studio / RViz 里就能看到一条"机器人打算往哪走"的预测路径线，调试导航和避障时极其直观——能直接看出"它为什么往这个方向偏"。

### 3.4 记最后速度与 `isStandingStill`（`robot_client.cpp:122`）

```cpp
_vx = x; _vy = y; _vtheta = theta;                 // 记住最后一条速度（近似当作当前速度）
_lastCmdTime = now();
if (fabs(_vx)>1e-3 || fabs(_vy)>1e-3 || fabs(_vtheta)>1e-3) _lastNonZeroCmdTime = now();
return call(CreateMoveMsg(x, y, theta));
```

记下最后一条速度和发令时刻。`isStandingStill`（`robot_client.cpp:151`）据此判停：

```cpp
bool isStandingStill(double timeBuffer) {
    return fabs(_vx)<1e-3 && fabs(_vy)<1e-3 && fabs(_vtheta)<1e-3
        && brain->msecsSince(_lastNonZeroCmdTime) > timeBuffer;   // 且已经停够久
}
```

> 💡 注意它判的是"**最后一条速度指令是 0 且距上次非零指令超过 timeBuffer**"，而非真去读关节速度。这是个轻量近似——下发 0 之后机器人还会惯性走一小段，所以要再等 `timeBuffer`（默认 1000ms）才认为"真停稳"。`RLVisionKick` 等需要"先站稳再动作"的逻辑会依赖它。

## 四、`crabWalk`：蟹步的方向补偿（`robot_client.cpp:130`）

`crabWalk(angle, speed)` 把"朝 `angle` 方向以 `speed` 蟹步"翻译成 `(vx, vy)`，供踢球节点冲球：

```cpp
double vxFactor = config->get_vx_factor();    // 补偿 x/y 实际速度比与参数不一致
double yawOffset = config->get_yaw_offset();  // 补偿定位角度偏差
double cmdAngle = angle + yawOffset;
double vx = cos(cmdAngle) * speed * vxFactor;
double vy = sin(cmdAngle) * speed;
if (fabs(vy) > vyLimit) {                      // 侧向超限→等比缩 vx，保持方向
    vx *= vyLimit / fabs(vy);
    vy = vyLimit * fabs(vy) / vy;
}
return setVelocity(vx, vy, 0);                 // vθ=0：蟹步不转身
```

> 💡 两个补偿参数解决"标称速度 ≠ 实际速度"的现实误差：`vx_factor` 修正机器人 x、y 两方向迈步效率不一样（前进往往比侧移快），让合速度方向真的指向 `angle`；`yaw_offset` 修正定位系统的固定角度偏差。冲球对方向精度极敏感（偏几度就踢歪），所以这两道补偿很关键。`crabWalk` 最终也走 `setVelocity`，因此前面三道安全处理对它同样生效。

## 五、`msecsToCollide`：防撞估算的坐标系修正（`robot_client.cpp:481`）

`msecsToCollide` 按当前速度向前推一条直线路径，估算"多少毫秒后会撞到其它机器人"，供避障逻辑用。把机体系速度 `(vx, vy)` 投到场地系时，之前的写法是错的：

```cpp
// 修正前（错误）：vx 只乘 cos、vy 只乘 sin，并非一次完整的二维旋转
// x0 + vx * cos(robotTheta) * maxTime / 1000,  y0 + vy * sin(robotTheta) * maxTime / 1000

// 修正后：对 (vx, vy) 做标准 SE(2) 旋转，得到场地系速度分量
const double vxField = vx * cos(robotTheta) - vy * sin(robotTheta);
const double vyField = vx * sin(robotTheta) + vy * cos(robotTheta);
Line path = {
    x0, y0,
    x0 + vxField * maxTime / 1000, y0 + vyField * maxTime / 1000
};
```

> 💡 机体系速度转场地系必须用完整旋转矩阵 `[cosθ −sinθ; sinθ cosθ]`。旧式只把 `vx` 乘 `cos`、`vy` 乘 `sin`，既漏了交叉项、`vy` 还用错了三角函数，算出的路径方向在机器人有侧移（`vy≠0`）或非零朝向时整体偏斜——防撞判断的那条预测线指错方向，可能把真正会撞的机器人判成"安全"。修正后 `vxField/vyField` 是正确的场地系速度，预测路径方向才准，避障才可靠。

## 小结

- 所有指令统一经 `call()`（`robot_client.cpp:16`）封成 `RpcReqMsg`（`api_id`→header、参数 JSON→body）发到 `LocoApiTopic{robot}Req`，**异步不等回执**（接口模式见 [模块02](../02-接口与消息/index.md)）。
- 指令表是一组薄封装：`moveHead`/`setVelocity`/`crabWalk`/`standUp`/`RLVisionKick`/`robocupWalk`/`enterDamping`/`waveHand`，只在 `api_id` 与 body 上不同。`moveHead` 对钳制后仍异常的角度打 `error`；`RLVisionKick` body 带 `version=2` 锁定行为；`robocupWalk` 切步态前先 `RLVisionKick(false)` 退视觉踢。
- `setVelocity`（`robot_client.cpp:83`）三道安全处理：**最小速度兜底 → 限速钳制 → 5 米路径仿真记录**，并记最后速度供 `isStandingStill`（`robot_client.cpp:151`）判停稳；新增 `applyMinX/Y/Theta` 开关可按轴关闭最小速度兜底。
- `crabWalk`（`robot_client.cpp:130`）用 `vx_factor`/`yaw_offset` 补偿迈步效率与定位偏差，让冲球方向更准，最终仍走 `setVelocity`。
- `msecsToCollide`（`robot_client.cpp:481`）修正机体系速度转场地系的旋转公式（补齐交叉项），使防撞预测路径方向正确。

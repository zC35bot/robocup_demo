# 4.3 · 队内通信 BrainCommunication

多个机器人组队踢球时，必须协调"谁去追球、谁助攻、守门员要不要压上"。这套机制由 `src/brain/src/brain_communication.cpp` 实现（基于 UDP），加上大脑里的 `handleCooperation`/`updateCostToKick` 做协作决策。本篇逐函数讲透。

---

## 一、整体架构：三通道 + 六线程

`BrainCommunication` 维护**三条逻辑通道**，每条都有"发送"和（部分有）"接收"线程：

| 通道 | 周期 | 作用 | 端口/地址 |
|------|------|------|-----------|
| 上报裁判机 | 1000ms | 发"我还活着"心跳，防被自动判失能 | 单播到 `game_control_ip` |
| 队友发现（广播）| 1000ms 发 + 持续收 | 互相发现队友的 IP | 组播 `239.255.255.250:20000+teamId` |
| 队内通信（单播）| 100ms 发 + 持续收 | 共享球位/角色/代价 | 单播 `30000+teamId` |

`initCommunication()`（`brain_communication.cpp:39`）是总入口，依次初始化六个部分：
```cpp
void BrainCommunication::initCommunication() {
    initGameControllerUnicast();   // 上报裁判机：建 socket + 起定时发送线程
    initDiscoveryBroadcast();      // 发现广播：起 1000ms 广播线程
    initDiscoveryReceiver();       // 发现接收：起阻塞收包线程
    initCommunicationUnicast();    // 队内单播：起 100ms 发送线程
    initCommunicationReceiver();   // 队内接收：起阻塞收包线程
}
```
> 💡 为什么发现用**组播**、通信用**单播**？发现阶段你还不知道队友在哪，只能往一个大家约定的组播地址喊"我在这"；一旦通过发现拿到了队友 IP，后续高频的状态同步就改用点对点单播，更高效。

每个线程都有原子布尔标志控制生命周期，析构时 `clearup*` 系列函数置位并 join，干净退出。

## 二、消息结构（`team_communication_msg.h`）

### 发现消息 `TeamDiscoveryMsg`
```cpp
struct TeamDiscoveryMsg {
    int validation = VALIDATION_DISCOVERY;  // 魔数 41203，校验"是不是自己人发的"
    int communicationId;                     // 递增序号
    int teamId;
    int playerId;
};
```
极简，只为"我是 X 队 Y 号，我在线"。

### 通信消息 `TeamCommunicationMsg`（核心）
每 100ms 给每个已发现队友单播一份"战况快报"：
```cpp
struct TeamCommunicationMsg {
    int validation = VALIDATION_COMMUNICATION;  // 魔数 31202
    int communicationId;
    int teamId, playerId;
    int playerRole;          // 1=前锋 2=守门员 3=未知
    bool isAlive;            // 在场且未被罚下
    bool isLead;             // 我处于"球权控制"状态（主攻）
    bool ballDetected;       // 我看到球了吗
    bool ballLocationKnown;
    double ballConfidence, ballRange;
    double cost;             // 我够到/踢到球的估计代价（约等于秒数）
    Point ballPosToField;    // 我看到的球在场地哪
    Pose2D robotPoseToField; // 我自己在场地哪
    double kickDir;          // 我打算往哪踢
    double thetaRb;          // 机器人-球角度
    int cmdId, cmd;          // 指令序号 + 编码指令（见下）
    uint64_t timestamp_ms;   // 发送时刻（用于丢弃过期包，Phase1新增）
    int packet_size;         // 包大小（Phase2 通信预算参考）
};
```

**`cmd` 的编码很巧妙**（注释在头文件里）：用十进制位编码两类指令：
- 百位=1（即 `cmd==100`）：我请求接管球权 → 收到的队友应让位。
- 十位=1 且个位存球员号（如 `cmd==011`）：守门员要出击，请求 1 号顶替当守门员。

## 三、发送侧三函数

### 上报裁判机 `unicastToGameController()`（`:237`）
```cpp
gc_return_data.team = brain->config->get_team_id();
gc_return_data.player = brain->config->get_player_id();
gc_return_data.message = GAMECONTROLLER_RETURN_MSG_ALIVE;  // 存活
sendto(gc_send_socket, &gc_return_data, ...);
```
每秒发一次 `HlRoboCupGameControlReturnData`（见 [4.2](./4.2-RoboCup协议常量.md)），告诉裁判机"我还在跑"，否则可能被自动罚下。

### 发现广播 `broadcastDiscovery()`（`:254`）
往组播地址发 `TeamDiscoveryMsg`，`communicationId` 每包递增。

### 队内单播 `unicastCommunication()`（`:365`）
把当前 `BrainData` 里的状态打包成 `TeamCommunicationMsg`，给**每个已发现的队友 IP** 各发一份。关键填充（`:377` 起）：
```cpp
msg.playerRole = (role=="striker")?1 : (role=="goal_keeper")?2 : 3;
msg.isLead = brain->data->tmImLead;
msg.cost = brain->data->tmMyCost;
msg.ballPosToField = brain->data->ball.posToField;
msg.cmd = brain->data->tmMyCmd;  // 我要发的指令
...
```

## 四、接收侧两函数

### 发现接收 `spinDiscoveryReceiver()`（`:276`）
绑定 `INADDR_ANY:20000+teamId`，阻塞 `recvfrom` 收包，逐个校验：
```cpp
if (recv.validation != VALIDATION_DISCOVERY) continue;  // 不是自己人
if (recv.teamId != myTeamId) continue;                  // 不是本队
if (recv.playerId == myPlayerId) continue;              // 是自己发的，忽略
// 记录：该队友 IP → playerId + 时间戳
```
把"队友 IP → playerId + 最近通信时间"存进映射表，作为单播目标。

`cleanupExpiredTeammates()`（`:327`）：**20 秒**没收到某队友的发现包就把它从映射表删掉（认为掉线）。

### 队内接收 `spinCommunicationReceiver()`（`:478`）
绑 `30000+teamId`，收 `TeamCommunicationMsg`，校验后把每个队友的状态存进 `brain->data->tmStatus[playerId-1]`，并记 `timeLastCom` 时间戳。这些就是协作决策的输入。

## 五、协作决策 `handleCooperation()`（`brain.cpp:577`）

每个 tick 调一次（见 [模块01](../01-启动与架构/index.md) 的 tick 流程）。它综合裁判机罚则和队友快报，决定**角色、谁主攻、要不要换守门员**。逐段：

### 1. 更新自身存活与代价
```cpp
data->tmImAlive = (penalty[selfIdx]==PENALTY_NONE) && odom_calibrated;  // 没被罚 且 定位成功
updateCostToKick();  // 算我够球的代价（见下节）
```
注意：定位没成功也算"不可信/不在线"——因为位置都不知道，信息没意义。

### 2. 统计在场队友
遍历 `tmStatus[]`，把"被裁判罚下"或"通信超时 5 秒"的队友标记为离线，收集在线队友索引。同时用裁判机数据统计 `gcAliveCount`（场上我方人数）。

### 3. 采信队友的球位
若我自己不知道球在哪，但某个在线队友报告看到球、且那球离我足够远（`tm_ball_dist_threshold`，避免采信明明在我跟前的球），就采信队友的球位写进 `data->tmBall` 并置黑板 `tm_ball_pos_reliable=true`（`brain.cpp:657`）。这让"我没看到球但队友看到了"时仍能往球的方向跑。

### 4. 角色自动切换
```cpp
if (!agentMode && enable_role_switch) {
    if (没被罚 && gcAliveCount < numOfPlayers) → 我必须当前锋;  // 场上缺人，别都守门
    if (被罚 && gcAliveCount == numOfPlayers-1) → 我必须当守门员; // 我是最后一个，最后的当守门
}
```

### 5. lead 竞争（带滞回，`brain.cpp:717`）
**核心防抖逻辑**。谁 cost 最低谁该主攻，但不能让两机因 cost 抖动来回让球：
```cpp
bool tmControllingBall = (有队友cost够低且比我低) || myCostRank>=2;
if (我当前是lead) {
    newLead = !tmControllingBall;       // 除非明显被超越，否则保持lead
} else {
    bool advantage = (myCostRank==0) && !tmControllingBall
                     && (没有现任lead || 我的cost比现任lead低一个margin);
    // 要抢lead：必须最优、且优势持续超过 lead_min_msec 才生效
    newLead = advantage 且 持续时间 > lead_min_msec;
}
```
> 💡 **滞回（hysteresis）** 是这里的关键：保持 lead 容易（只要没被明显超越），抢 lead 难（要明显更优且持续一段时间）。这就避免了"两个机器人 cost 差不多、每帧轮流当主攻、谁也不真正去踢"的灾难。`handover_margin_m` 是优势阈值，`lead_min_msec` 是持续时间门槛。

### 6. 守门员换人
如果我是守门员且是 lead（说明我离球最近该去踢），但我离自家球门太远，就发 `cmd = 10 + 最近球门的队友id`，请那个队友来当守门员，我自己压上进攻（`brain.cpp:780`）。

### 7. 处理收到的指令（`brain.cpp:793`）
```cpp
if (cmd == 100) { tmImLead = false; }              // 队友要主攻，我让位助攻
else if (cmd>10 && cmd<20) {                        // 守门员要换人
    if (cmd-10 == selfId) → 我变守门员;
}
data->tmReceivedCmd = 0;  // 处理一次就清零
```

## 六、代价函数 `updateCostToKick()`（`brain.cpp:1069`）

cost 是协作的"货币"——cost 越低越该去踢。它把多个因素累加（每项都打 debug 日志）：

```cpp
cost += 球多久没看到（秒）;
if (!ball_location_known)  cost += 5.0;        // 球丢了
cost += data->ball.range;                       // 到球距离
cost += 0.2 * range / (ballSpeed + 0.5);        // 球在动更难拦截（Phase1速度项）
if (distToObstacle(球方向) < 1.5) cost += 0.5;  // 去球路上有障碍
cost += fabs(ball.yawToRobot);                   // 转向对准球的角度代价
// 会撞到某个更近的队友？ cost += 2.0
cost += fabs(kickDir - robotBallAngle) * 0.4/0.3; // 绕球调整角度的代价
if (recoveryState == HAS_FALLEN) cost += 15.0;   // 摔倒了
if (!odom_calibrated) cost += 100;               // 定位都没有，基本别去抢
data->tmMyCost = lastCost*0.8 + cost*0.2;        // 指数平滑防抖
```
> 💡 各项权重体现了优先级直觉：定位失败(+100)是"一票否决"，摔倒(+15)和球丢(+5)是重罚，距离/角度是连续成本。最后的**指数平滑**让 cost 不会因单帧抖动剧烈跳变，配合 lead 滞回共同保证协作稳定。

cost 算完存进 `data->tmMyCost`，既用于本机的 lead 判断，也通过 `TeamCommunicationMsg` 广播给队友，让全队达成"谁去踢"的分布式共识。

## 七、🏆 规则红线（务必注意）

最新 RoboCup 规则**禁止机器人之间的单播通信**，并限制通信预算（包数量/包大小）。本文件实现的**队内单播通道（第四节）尚未适配**这条新规。根目录 README 明确写道：

> 若比赛需要机器人通信，需修改 `brain_communication.cpp` 以符合规则。

`TeamCommunicationMsg` 里 Phase1 新增的 `packet_size` 字段就是为将来做"通信预算统计"预留的。正式参赛前必须改造这部分（例如改成组播、压缩包、降频）。

---

## 小结

- 三通道（上报裁判机/发现广播/队内单播）+ 六线程，原子标志管理生命周期。
- 发现用组播找队友、通信用单播高频同步状态。
- `handleCooperation` 用 **cost 排序 + lead 滞回** 决定谁主攻，避免抢球抖动。
- `updateCostToKick` 把距离/角度/障碍/摔倒/定位/球速等因素加权成一个 cost，再平滑。
- 🏆 单播通道未适配最新禁单播规则，参赛需改。

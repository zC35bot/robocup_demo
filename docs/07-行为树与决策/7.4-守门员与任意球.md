# 7.4 · 守门员与任意球

本篇讲两块：① 守门员的决策（`GoalieDecide`）和它的封堵站位；② 任意球的整套就位逻辑，以及背后几个由 C++ 维护的关键标志（开球计时、等对方开球、球出界）。

源码：`brain_tree.cpp:545/1046`、`brain.cpp:542/985/1181/1206`、`subtree_goal_keeper_play.xml`、`subtree_striker_freekick.xml`、`subtree_goal_keeper_freekick.xml`。

---

## 一、GoalieDecide：守门员决策（`brain_tree.cpp:1046`）

守门员比前锋**保守得多**——主要任务是守在门前、把来球挡住或清出去，而不是追到天涯海角。它输出 5 种 `decision`，仍是优先级 if-else（`brain_tree.cpp:1066-1084`）：

```
①  不知道球在哪              → "find"
②  球在我方半场以内一点      → "retreat"   （退回门前封堵）
③  球还远（ballRange>阈值）  → "chase"     （阈值默认 3.0，比前锋大）
④  对准了（身前半圆）        → "kick"      （清球）
⑤  否则                      → "adjust"
```

### 1.1 retreat：守门员的核心（`brain_tree.cpp:1070`）

```cpp
else if (brain->data->ball.posToField.x > 0 - static_cast<double>(lastDecision == "retreat"))
    newDecision = "retreat";
```

- 判据是 `ball.posToField.x > 0`（球在中线偏我方一侧之外，即球在对方半场或中场偏远）——这时守门员不该冲出去，而是**退回门前封堵**（`GoToGoalBlockingPosition`）。
- `- static_cast<double>(lastDecision=="retreat")` 这个写法是**滞回**：`lastDecision=="retreat"` 时 `(double)true=1`，门槛变成 `x > -1`；否则 `x > 0`。即"一旦开始退守，就把放行边界往本方挪 1m"，避免球在中线附近来回时守门员"退一下→冲一下"反复横跳。

> 💡 守门员决策和前锋共用同样的滞回套路（`chase` 也有 `*0.9`，`brain_tree.cpp:1073`），但 `chase_threshold` 默认是 **3.0**（前锋是 1.0，见 XML）。意思是守门员要等球进到 3m 内才考虑亲自上去抢，否则老老实实在门前横移。这符合守门员"不轻易出击"的定位。

### 1.2 守门员的 kickDir：清球而非射门（`brain_tree.cpp:1054`）

```cpp
double kickDir = atan2(ball.posToField.y, ball.posToField.x + length/2);
```

注意守门员**自己算 kickDir**（不走 `CalcKickDir` 的 shoot 逻辑）：方向是"从本方球门中心指向球"的延长线，即**朝远离本方球门的方向把球清出去**。`angleIsGood` 只要求 `dir_rb_f ∈ (-π/2, π/2)`（球在身前半圆即可踢），比前锋的对准要求宽松——守门员清球图的是"把球弄走"，不追求精准。

## 二、GoalKeeperPlay 子树（`subtree_goal_keeper_play.xml`）

结构和 `StrikerPlay`（[7.3](./7.3-前锋决策.md)）几乎对称，差别在核心动作排：

```xml
<Sequence>
   <SubTree ID="CamFindAndTrackBall" _autoremap="true" />
   <GoalieDecide decision_out="{decision}" decision_in="{decision}" chase_threshold="3.0"/>
   <GoToGoalBlockingPosition _while="decision=='find'"    theta_tolerance="0.6" />
   <GoToGoalBlockingPosition _while="decision=='retreat'" theta_tolerance="0.6" />
   <Chase  _while="decision=='chase'"  vx_limit="0.8" .../>
   <Adjust _while="decision=='adjust'" .../>
   <Kick   _while="decision=='kick'"   speed_limit="1.2" min_msec_kick="600"/>
</Sequence>
```

- **没有 `KickSelector`**——守门员不需要精细射门选择，`GoalieDecide` 一层就够。
- `find` 和 `retreat` **都走 `GoToGoalBlockingPosition`**：找不到球时也先退回门前守着（总比乱跑强）。
- `Kick` 参数更猛：`speed_limit="1.2"`（前锋普通踢 0.8）、`min_msec_kick="600"`（更短）——守门员清球求快求狠。

> 💡 守门员把"找不到球"也映射到"回门前站位"，体现了保守策略：守门员的默认状态就是"守在门口"，任何不确定都退回这个安全位。

## 三、GoToGoalBlockingPosition：门前封堵几何（`brain_tree.cpp:545`）

这是守门员（和前锋退守时）的封堵站位算法。核心思想：**站在"球→本方球门"连线上、距门线 `distToGoalline` 处**，挡住射门路线。

```cpp
// x：站在门线前 distToGoalline 处
targetPose.x = curRole=="striker" ? max(-length/2 + distToGoalline, ballPos.x - 1.5)
                                  : (-length/2 + distToGoalline);
// y：把球与门连线投影到门线前那一点
if (ballPos.x + length/2 < distToGoalline) {        // 球已贴近门线
    targetPose.y = curRole=="striker" ? (ballPos.y>0 ? goalWidth/2 : -goalWidth/2)
                                      : (ballPos.y>0 ? goalWidth/4 : -goalWidth/4);
} else {
    targetPose.y = ballPos.y * distToGoalline / (ballPos.x + length/2);   // 相似三角形投影
    targetPose.y = curRole=="striker" ? cap(.., goalWidth/2, -goalWidth/2)
                                      : cap(.., penaltyAreaWidth/2, -penaltyAreaWidth/2);
}
```

- **y 的几何**：`ballPos.y * distToGoalline / (ballPos.x + length/2)` 是相似三角形——把球相对本方门中心的方向，投影到"门前 `distToGoalline`"这条线上的 y 坐标。球偏左守门员就偏左站，始终堵在球与门之间。
- **striker vs goal_keeper 不同**：守门员 x 固定贴门线（`-length/2 + distToGoalline`），y 限幅在禁区宽度内、贴近门线时只走到门宽 1/4（守得更紧）；前锋退守时 x 可以离球近一点（`ballPos.x - 1.5`），y 限幅到门宽（站得更靠外）。

> 🏆 这就是真实守门员的"封角"——永远把身体保持在球和球门之间。投影到门线前一小段距离（而不是站在门线上）是为了能更早碰到球。前锋退守用更宽的限幅，因为它是临时帮忙补防，不必死守门线。

## 四、任意球：两个子树 + 就位逻辑

[7.2](./7.2-主树game_xml.md) 讲过任意球在 `GET_READY` 阶段按角色进 `StrikerFreekick` / `GoalKeeperFreekick`。

### 4.1 StrikerFreekick（`subtree_striker_freekick.xml`）

按 `gc_is_sub_state_kickoff_side`（我方是否执行任意球方）分进攻/防守两支，结构对称：

```xml
<ReactiveSequence _while="gc_is_sub_state_kickoff_side">   <!-- 进攻 -->
   <CamTrackBall _while="ball_location_known" />
   <CalcKickDir />
   <SubTree ID="FindBall" _while="!(ball_location_known||tm_ball_pos_reliable)" />
   <GoToFreekickPosition _while="ball_location_known||tm_ball_pos_reliable" side="attack" vx_limit="0.7" vy_limit="0.4" />
</ReactiveSequence>
<ReactiveSequence _while="!gc_is_sub_state_kickoff_side">  <!-- 防守，side="defense" -->
   ...
</ReactiveSequence>
```

### 4.2 GoToFreekickPosition：按角色排名站位（`brain_tree.cpp:432`）

这是 `StatefulActionNode`（跨帧导航）。它按 `side`（attack/defense）和 `myStrikerIDRank`（本机在前锋里的编号 0~3）算各自的站位（`brain_tree.cpp:450-485`）：

| Rank | attack 站位 | defense 站位 |
|------|-------------|--------------|
| 0 | 球后方 `dist` 处、朝 `kickDir`（**主罚者**，正对踢球方向） | 球后 3m、偏防守方向（离球更远） |
| 1 | 球后 2m、偏 `defenseDir`（侧应） | 球后 3.5m、更偏侧 |
| 2 | 禁区前 `penaltyDist` 处、`+goalAreaWidth/2`（站位补防） | 同 attack |
| 3 | 禁区前 `penaltyDist` 处、`-goalAreaWidth/2` | 同 attack |

> 🏆 任意球规则要求非执行方退到离球一定距离外。`myStrikerIDRank==0` 的主罚者正对 `kickDir` 站在球后准备开球，其余队友散开到接应/补防位。防守时所有人都退得更远（rank0 从 2m 改到 3m），符合"防守方必须远离球"的规则。

到位判定：`dist<0.2 && |deltaDir|<0.1` 就 `setVelocity(0,0,0)` 返回 `SUCCESS`。导航分两段：远（>1m 且开避障）用底层 `moveToPoseOnField3` 带避障；近了（`_isInFinalAdjust`）改用细调速度，并在路径会撞球时（`pointMinDistToLine<0.5 && range<1.0`）把 vx 压 0、vy 加 0.1 绕开球。

### 4.3 GoalKeeperFreekick（`subtree_goal_keeper_freekick.xml`）

守门员任意球简单：

```xml
<ReactiveSequence>
   <SubTree ID="CamFindAndTrackBall" _autoremap="true" />
   <GoToReadyPosition         _while="!ball_location_known" />    <!-- 找不到球→回起始位 -->
   <GoToGoalBlockingPosition  _while="ball_location_known" />      <!-- 知道球→门前封堵 -->
   <SelfLocate mode="trust_direction" _while="odom_calibrated" />
</ReactiveSequence>
```

不管是谁的任意球，守门员都老老实实回门前封堵（知道球位）或回起始位（不知道）。

## 五、handleSpecialStates：开球计时与庆祝（`brain.cpp:542`）

每个心跳调一次，维护几个跨帧标志（`KICKOFF_DURATION = 10` 秒）：

```cpp
// 开球计时
if (gameState=="SET" && isKickoffSide) { isKickingOff=true; kickoffStartTime=now; }
else if (msecsSince(kickoffStartTime) > 10*1000) isKickingOff=false;

// 任意球开球计时（brain.cpp:559）
if (gameState=="PLAY" && subType=="FREE_KICK" && isFreekickKickoffSide) {
    isFreekickKickingOff=true; freekickKickoffStartTime=now;
} else {                                          // 已离开"我方任意球开球"状态
    if (msecsSince(freekickKickoffStartTime) > 10*1000)
        isFreekickKickingOff=false;               // 开球窗口仍按 10 秒到点才清
    isDirectShoot=false;                          // 直接射标志：每帧无条件复位
}

// 进球庆祝
if (score > lastScore) { setEntry("we_just_scored", true); lastScore=score; }
if (gameState=="SET")  setEntry("we_just_scored", false);
```

- **`isKickingOff` / `isFreekickKickingOff`**：标记"我方正处于开球/任意球开球后的 10 秒窗口"。这两个标志影响踢球判据——比如 `StrikerDecide` 的踢球分支里 `!isFreekickKickingOff` 会在任意球开球期间禁用普通 angle-good 踢球（`brain_tree.cpp:887`），逼机器人用专门的开球逻辑。
- **`isDirectShoot`**：标记"当前任意球是可直接射门的类型"——`gameControlCallback` 把 `DIRECT_FREEKICK`/`PENALTY_KICK`/`GOAL_KICK` 三种子状态置 `isDirectShoot=true`（`brain.cpp:1488/1499/1510`），其余任意球（间接、角球、掷界外球等）保持 `false`，用于日志与后续直射逻辑区分。
- **`we_just_scored`**：进球瞬间（`score` 增加）置真，进 SET（下一回合开始）时清零。用于触发挥手庆祝（`WaveHand`，见 [7.6](./7.6-找球与移动节点.md)）。

> 💡 **安全加固：`isDirectShoot` 的复位从 10 秒计时里独立出来**（`brain.cpp:562`）。旧代码把 `isDirectShoot=false` 塞在 `else if (msecsSince > 10s)` 分支里，导致它必须**等满 10 秒**才清——若比赛已切回 `PLAY` 常规进攻，这个"可直接射门"标志会**残留(stale)一段时间**，误导日志/直射判据。现在改成：只要不在"我方任意球开球"触发态，就**每帧无条件** `isDirectShoot=false`（`isFreekickKickingOff` 仍保留 10 秒窗口逻辑）。这样任意球一结束，直射标志立刻失效，不会泄漏进 open play。

> 🏆 RoboCup 开球后有一段"球必须先移动一定距离/经过若干秒才算活球"的规则期。10 秒计时让机器人知道自己还在这个特殊窗口里，避免过早做出违规的踢球/抢球。

## 六、wait_for_opponent_kickoff：等对方开球（`brain.cpp:985`）

`updateKickoffMemory` 维护 `wait_for_opponent_kickoff` 标志——对方开球时，我方必须退开、不得碰球，直到球动了或超时：

```cpp
bool isWaitingForKickoff = (gameState=="SET"||gameState=="READY") && !gc_is_kickoff_side;
bool isWaitingForFreekick = (subState=="SET"||subState=="GET_READY") && !gc_is_sub_state_kickoff_side;
if (isWaitingForFreekick || isWaitingForKickoff) {
    ballPos = ball.posToRobot;            // 记下球此刻位置
    kickOffTime = now();
    setEntry("wait_for_opponent_kickoff", true);
} else if (getEntry("wait_for_opponent_kickoff")) {
    if (ballMoved() || timeReached())     // 球动了 或 超 10 秒
        setEntry("wait_for_opponent_kickoff", false);   // 解除
}
```

- `ballMoved()`：球相对开球时记录位置的位移超过 `max(range*0.15, 0.3)` 米，认为对方已开球。
- `timeReached()`：超 10 秒强制解除（防止对方迟迟不开球或我方没看到球动而永远干等）。

标志为真时，[7.3](./7.3-前锋决策.md) 里 `StrikerPlay` 子树就只 `CamFindAndTrackBall` + `SetVelocity`（盯球但身体不动）。

> 🏆 这直接对应规则："对方开球时，我方不得进入中圈、不得先碰球。"机器人记下球的初始位置，一旦观测到球移动（对方已开球），立刻解除限制投入比赛——既守规则又不浪费一秒反应时间。

## 七、isBallOut / updateBallOut：球出界判定（`brain.cpp:1181/1206`）

`ball_out` 标志触发 `StrikerPlay`/`GoalKeeperPlay` 里的"回场"分支（[7.3](./7.3-前锋决策.md)）。两种判据（`isBallOut`，`brain.cpp:1181`）：

1. **定位判据**：球的场地坐标超出场地边界 + 余量（`|x|>length/2+d` 或 `|y|>width/2+d`）。
2. **线判据**：看到可信的边线/球门线（`confidence>1.0`）时，球到该线的**垂直距离** `> lineCompareDist` 也算出界。

`updateBallOut`（`brain.cpp:1206`）在外面加了**滞回放宽**：

```cpp
double range = lastBallOut ? 4.0 : 2.5;              // 出界后放宽可信球距
double threshold = get_ball_out_threshold();
threshold += isFreekickKickingOff ? 1.0 : 0.0;       // 任意球期间放宽
threshold *= lastBallOut ? 1.0 : 1.5;                // 上次没出界→门槛×1.5，防抖
setEntry("ball_out", isBallOut(threshold, 10.0) && ball.range < range);
```

> 💡 三处放宽都是为了**防止误判和抖动**：① `lastBallOut?1.0:1.5` 让"判出界"比"判回场"门槛高，避免球在边线附近反复出界/回场；② 任意球期间 +1.0 更宽容（任意球球本就常贴边线）；③ `ball.range < range` 要求球离自己够近才信——远处的球定位误差大，不轻易判出界。`isBallOut` 还要求看到**高置信度**的边线（`>1.0`）才用线判据，进一步防止误报。

## 小结

- `GoalieDecide`（`brain_tree.cpp:1046`）更保守：find → retreat → chase(阈值 3.0) → kick → adjust，`retreat` 用 `x>0` 带 1m 滞回，守门员自算朝门外的清球 `kickDir`。
- `GoToGoalBlockingPosition`（`brain_tree.cpp:545`）用相似三角形把球投影到门线前 `distToGoalline` 处，始终堵在球与门之间；striker/goal_keeper 限幅不同。
- 任意球按 `gc_is_sub_state_kickoff_side` 分进攻/防守，`GoToFreekickPosition`（`brain_tree.cpp:432`）按 `myStrikerIDRank` 给每个前锋排专属站位；守门员一律回门前封堵。
- `handleSpecialStates`（`brain.cpp:542`）维护开球 10 秒计时（`isKickingOff`/`isFreekickKickingOff`）和进球庆祝（`we_just_scored`）；`isDirectShoot` 的复位已从 10 秒计时里独立出来，离开任意球态即每帧清零，避免直射标志残留进常规进攻。
- `wait_for_opponent_kickoff`（`brain.cpp:985`）让我方在对方开球时只看球不动，球动或超 10 秒才解除；`ball_out`（`brain.cpp:1206`）用定位+边线两种判据并加滞回防抖。

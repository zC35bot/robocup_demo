# 7.1 · 行为树框架与黑板

本篇把 `BehaviorTree.CPP` 的核心概念、`BrainTree` 怎么把 C++ 节点注册进框架、黑板里每一个默认变量的含义讲清楚。读完它，后面几篇看 XML 和节点代码就不会卡。

源码：`src/brain/include/brain_tree.h`、`src/brain/src/brain_tree.cpp`。

---

## 一、BehaviorTree.CPP 基本概念

### 1.1 节点的三种返回状态

行为树的每个节点被 **tick（敲一下）** 时，必须返回三种状态之一：

| 状态 | 含义 | 典型场景 |
|------|------|----------|
| `SUCCESS` | 这次干完了/条件成立 | 走到了目标点、条件判断为真 |
| `FAILURE` | 失败/条件不成立 | 走不到、条件为假 |
| `RUNNING` | 还在进行中，下一帧继续 | 正在走路、踢球动作还没结束 |

> 💡 `RUNNING` 是行为树相对状态机最关键的优势：一个动作可以**跨很多帧**执行（比如踢球持续 1 秒），节点返回 `RUNNING` 占住执行权，下一帧从它继续，而不必把"踢球进行到第几毫秒"这种状态手动塞进全局变量。

### 1.2 控制节点：决定子节点怎么跑

本项目主要用到这几种控制节点：

- **`Sequence`（顺序）**：从左到右依次 tick 子节点。子节点返回 `SUCCESS` 就继续下一个；返回 `FAILURE` 则整个 Sequence 立刻 `FAILURE`；返回 `RUNNING` 则 Sequence 也 `RUNNING` 并**记住执行到第几个**，下一帧从那个子节点继续（不会重头评估前面的）。
- **`ReactiveSequence`（反应式顺序）**：和 Sequence 类似，但**每次 tick 都从头重新评估所有子节点**。只要前面任何一个子节点变成 `FAILURE`，立刻中断后面的。这正是"实时响应条件变化"的关键——见下文 `_while`。
- **`IfThenElse`**：三个子节点 = 条件 / Then 分支 / Else 分支。条件 `SUCCESS` 走 Then，`FAILURE` 走 Else。本项目在 `subtree_cam_find_and_track_ball.xml` 里用它实现"知道球位 → 追踪 / 不知道 → 找球"。
- **`RunOnce`**：包住的子节点只在第一次 tick 时执行一次，之后直接放行。`game.xml` 用它初始化 `control_state=3`。

> 💡 **Sequence vs ReactiveSequence 的区别为什么重要？** 假设序列是"看球 → 决策 → 执行动作"。如果用普通 `Sequence`，当"执行动作"返回 `RUNNING`（踢球进行中），下一帧会**直接从"执行动作"继续**，跳过了"看球/决策"——一旦此时裁判吹停、或球被对手抢走，机器人还在埋头踢空气。而 `ReactiveSequence` 每帧都从"看球"重新评估，比赛状态一变、决策一变，立刻就能切走。本项目主树和两个 Play 子树**几乎全是 `ReactiveSequence`**，就是为了这种"任何时刻都能被打断"的实时性。

### 1.3 `_while` 语法糖

XML 节点上写 `_while="条件表达式"` 是 `BehaviorTree.CPP` 的语法糖：**只有当条件成立时才执行该节点；条件不成立时直接放行（视作不阻塞）**。条件表达式可以直接读黑板变量，例如：

```xml
<SubTree ID="StrikerPlay" _while="player_role == 'striker'" />
<Chase _while="decision == 'chase'" .../>
<Sequence _while="ball_out"> ... </Sequence>
```

配合 `ReactiveSequence`（每帧重判），`_while` 就让"一排动作节点里只有 `decision` 命中的那个执行"成为可能——这正是 [README](./index.md) 讲的**决策-动作分离**的实现机制。

### 1.4 `Script` / `ScriptCondition`：在 XML 里写小逻辑

- `<Script code="control_state=3" />`：直接给黑板变量赋值。
- `<ScriptCondition code="ball_location_known || tm_ball_pos_reliable" />`：把表达式求值当条件用，真→`SUCCESS`，假→`FAILURE`。

## 二、`BrainTree::init()`：把 C++ 节点注册进框架（`brain_tree.cpp:26`）

XML 里写的 `<Chase>`、`<StrikerDecide>` 这些标签，框架并不天生认识——必须先把"标签名 → 对应 C++ 类"的映射注册进 `BehaviorTreeFactory`。代码用了一个宏来减少重复：

```cpp
// brain_tree.cpp:21
#define REGISTER_BUILDER(Name)     \
    factory.registerBuilder<Name>( \
        #Name,                     \
        [this](const string &name, const NodeConfig &config) { return make_unique<Name>(name, config, brain); });
```

> 💡 这个宏展开后，`REGISTER_BUILDER(Chase)` 等价于"告诉工厂：遇到名为 `Chase` 的 XML 标签，就 `new Chase(name, config, brain)`"。**注意每个节点的构造都注入了 `brain` 指针**——这是节点访问全局世界状态（`brain->data`）、配置（`brain->config`）、底层运动接口（`brain->client`）、黑板（`brain->tree`）的唯一通道。这就是为什么节点代码里到处是 `brain->...`。

`init()` 的完整流程（`brain_tree.cpp:26`）：

```cpp
void BrainTree::init() {
    BehaviorTreeFactory factory;
    REGISTER_BUILDER(RobotFindBall)   // ① 注册所有动作/决策节点
    REGISTER_BUILDER(Chase)
    REGISTER_BUILDER(SimpleChase)
    ... // 共 26 个常规节点
    brain->registerLocatorNodes(factory);          // ② 注册定位相关节点（见模块06）
    REGISTER_BUILDER(CalibrateOdom)                // ③ 调试用节点
    REGISTER_BUILDER(PrintMsg)
    factory.registerBehaviorTreeFromFile(brain->config->get_tree_file_path()); // ④ 从文件加载 XML
    tree = factory.createTree("MainTree");          // ⑤ 实例化 MainTree
    initEntry();                                    // ⑥ 初始化黑板默认值
}
```

四步要点：
1. **②** 定位节点（`SelfLocate`、`SelfLocate1M`、`Locate` 子树用到的那些）由 `registerLocatorNodes` 单独注册，逻辑属于 [模块06](../06-定位与球预测/index.md)。
2. **④** `get_tree_file_path()` 返回哪个 XML（默认 `game.xml`）。XML 里的 `<include>` 会把所有子树文件一并加载进工厂。
3. **⑤** `createTree("MainTree")` 从注册表里找名为 `MainTree` 的 `BehaviorTree` 当根，递归把 `<SubTree>` 展开成一棵完整的树。
4. **⑥** 树建好后立刻调 `initEntry()` 给黑板填默认值——**否则第一帧节点去 `getEntry` 还没写过的 key 会拿到空值**。

`tick()` 极简（`brain_tree.cpp:122`）：

```cpp
void BrainTree::tick() { tree.tickOnce(); }
```

每个心跳由 `Brain::tick`（[模块01 · 1.4](../01-启动与架构/1.4-main与多线程.md)）调用它一次，从根往下走一遍。

## 三、黑板：节点间的共享变量（`getEntry` / `setEntry`）

黑板是整棵树共享的键值表。`BrainTree` 提供了两个模板方法（`brain_tree.h:25/34`）：

```cpp
template <typename T>
inline T getEntry(const string &key) {           // 读：key 不存在时返回该类型默认值（value{}）
    T value{};
    auto res = tree.rootBlackboard()->get<T>(key, value);
    return value;
}
template <typename T>
inline void setEntry(const string &key, const T &value) {  // 写
    tree.rootBlackboard()->set<T>(key, value);
}
```

> 💡 注意 `getEntry` 用的是 **`rootBlackboard()`**——根黑板。子树默认有自己的局部黑板，但 XML 里几乎所有 `<SubTree>` 都写了 `_autoremap="true"`，意思是"把同名变量自动映射到父黑板"，于是全树共享同一组变量。`brain->tree->getEntry<bool>("ball_out")` 在任何节点里读到的都是同一个 `ball_out`。

谁来写黑板？两类：
- **C++ 感知/回调代码**：大脑每帧把世界状态同步进黑板。例如 `gameControlCallback`（`brain.cpp:1414`）写 `gc_game_state`；定位/球处理逻辑写 `ball_location_known`、`ball_out`、`pred300_valid` 等。
- **决策节点**：`StrikerDecide`/`KickSelector` 等用 `setOutput("decision_out", ...)` 写 `decision`（XML 里通过 `decision_out="{decision}"` 把端口连到黑板的 `decision`）。

## 四、`initEntry()`：黑板每个变量的含义（`brain_tree.cpp:73`）

这是理解整套决策的"字典"。下表逐个解释 `initEntry` 里设的默认值（行号对应 `brain_tree.cpp`）：

### 4.1 角色与球感知

| 变量 | 默认 | 含义 |
|------|------|------|
| `player_role` | 配置值 | 本机角色 `striker` / `goal_keeper`，主树据此选子树 |
| `ball_location_known` | `false` | **我自己**是否知道球的可靠位置（不是看到，而是"记得"） |
| `tm_ball_pos_reliable` | `false` | 队友报的球位是否可信（自己看不到时的备份，见 [模块04](../04-裁判机与通信/index.md)） |
| `ball_out` | `false` | 球是否出界（`updateBallOut`，见 [7.4](./7.4-守门员与任意球.md)） |
| `track_ball` | `true` | 是否让头部追踪球 |
| `ball_range` | `0` | 球到机器人的距离缓存 |

### 4.2 决策结果

| 变量 | 默认 | 含义 |
|------|------|------|
| `decision` | `""` | **核心**：当前决策字符串（`chase`/`adjust`/`kick`/`shoot`/`find`/`assist`/...），决策节点写、动作节点读 |
| `defend_decision` | `"chase"` | 防守决策缓存 |

### 4.3 裁判机状态（由 `gameControlCallback` 持续刷新，见 [7.2](./7.2-主树game_xml.md)）

| 变量 | 默认 | 含义 |
|------|------|------|
| `gamecontroller_isKickOff` | `true` | 是否开球 |
| `gc_game_state` | `""` | 主状态字符串：`INITIAL`/`READY`/`SET`/`PLAY`/`END` |
| `gc_game_sub_state_type` | `"NONE"` | 副状态类型：`NONE`/`FREE_KICK`/`TIMEOUT` |
| `gc_game_sub_state` | `""` | 副状态阶段：`STOP`/`GET_READY`/`SET` |
| `gc_is_kickoff_side` | `false` | 我方是否开球方 |
| `gc_is_sub_state_kickoff_side` | `false` | 任意球等副状态中，我方是否执行方 |
| `gc_is_under_penalty` | `false` | 我是否正被罚下/换人中 |

### 4.4 协作与控制

| 变量 | 默认 | 含义 |
|------|------|------|
| `is_lead` | `true` | 我是否主攻（lead）。协作逻辑写，决策读（`!tmImLead → assist`） |
| `goalie_mode` | `"attack"` | 守门员模式 |

| `assist_chase` / `assist_kick` | `false` | 手柄辅助追球/踢球开关 |
| `go_manual` | `false` | 是否进入手动模式 |
| `odom_calibrated` | `false` | 里程计/定位是否已标定（很多动作的前提） |
| `test_choice` | `0` | 测试用选择 |

### 4.5 庆祝 / 开球 / 球预测（Phase1）

| 变量 | 默认 | 含义 |
|------|------|------|
| `we_just_scored` | `false` | 刚进球（用于挥手庆祝，`handleSpecialStates` 维护） |
| `wait_for_opponent_kickoff` | `false` | 是否在等对方开球（此时只看球不动） |

| `ball_prediction_valid` | `false` | 球预测总开关（`Chase` 用 pred100） |

### 4.6 自动视觉标定（`CalibrateOdom` / 视觉相关，了解即可）

`calibrate_state`、`calibrate_pitch_center/step`、`calibrate_yaw_center/step`、`calibrate_z_center/step` —— 一组用于自动校准相机俯仰/偏航/高度的中间变量，和决策主流程关系不大。

## 五、`SyncActionNode` vs `StatefulActionNode`

本项目的节点继承两种基类，区别在于"能不能跨帧"：

| | `SyncActionNode` | `StatefulActionNode` |
|---|---|---|
| 实现方法 | 只实现 `tick()` | 实现 `onStart()` / `onRunning()` / `onHalted()` |
| 返回 | 一般每帧 `SUCCESS`（或 `FAILURE`） | `onStart` 返回 `RUNNING` 进入持续态，`onRunning` 每帧续，干完返回 `SUCCESS` |
| 适用 | 瞬间算完的事：决策、设速度、看头一眼 | 跨多帧的事：踢球、转身、走到指定点 |
| 被打断 | 直接不再被 tick | 框架会调 `onHalted()` 做收尾 |

举例：
- `SetVelocity`、`StrikerDecide`、`Chase`、`Adjust` 是 **`SyncActionNode`**——每帧瞬间算完速度/决策（注意 `Chase`/`Adjust` 虽是连续动作，但每帧只设一次速度就返回 `SUCCESS`，靠外层 `ReactiveSequence` 每帧重进来）。
- `Kick`、`RLVisionKick`、`StandStill`、`TurnOnSpot`、`RobotFindBall`、`CamFastScan`、`GoToFreekickPosition` 是 **`StatefulActionNode`**——踢球/转身需要持续好几帧，用 `onRunning` 占住执行权，并在 `onHalted` 里处理"被更高优先级决策打断"时的收尾（如 `Kick::onHalted` 重置计时器）。

> 💡 **为什么 `Chase`/`Adjust` 是 Sync 而 `Kick` 是 Stateful？** 因为追球/绕球调整本身是"无记忆"的——每帧根据当前球位重新算一次速度即可，被打断也无需收尾。而踢球是一个"启动→稳定→蟹步冲球→结束"的有头有尾的过程，中途被打断必须把状态机/计时器复位，否则下次进来会错乱。所以踢球类用 Stateful。详见 [7.5](./7.5-动作节点-追球调整踢球.md)。

## 小结

- `BehaviorTree.CPP` 节点 tick 返回 `SUCCESS/FAILURE/RUNNING`；`ReactiveSequence` 每帧重判，配合 `_while` 实现"实时可打断"和"决策-动作分离"。
- `BrainTree::init`（`brain_tree.cpp:26`）用 `REGISTER_BUILDER` 宏把 26 个节点 + 定位节点注册进工厂，加载 `game.xml`，建 `MainTree`，再 `initEntry` 填黑板默认值。每个节点都注入 `brain` 指针访问全局。
- 黑板用 `getEntry/setEntry` 读写**根黑板**（`_autoremap` 让子树共享）；`initEntry`（`brain_tree.cpp:73`）定义了角色、球感知、裁判机状态、协作、球预测等全部共享变量。
- `SyncActionNode`（瞬时：决策/设速度）与 `StatefulActionNode`（跨帧：踢球/转身，有 `onStart/onRunning/onHalted`）是节点的两种基类。

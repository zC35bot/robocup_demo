# 6.2 · 定位行为树节点

[6.1](./6.1-自定位Locator.md) 讲的 `locateRobot` 只是个算法。真正在比赛中**何时定位、用什么约束、成功后落地**，由一组行为树节点完成。本篇讲这些 `SelfLocate*` 节点和它们如何被 `Locate` 子树编排。

---

## 一、节点注册（`locator.cpp:13`）

`RegisterLocatorNodes` 用宏把 8 个定位节点注册进行为树工厂：
```cpp
REGISTER_LOCATOR_BUILDER(SelfLocate);           // 通用，带模式
REGISTER_LOCATOR_BUILDER(SelfLocateEnterField);  // 入场快速定位
REGISTER_LOCATOR_BUILDER(SelfLocate1M);          // 单角点微调
REGISTER_LOCATOR_BUILDER(SelfLocateBorder);      // 靠边界线
REGISTER_LOCATOR_BUILDER(SelfLocate2T);          // 一对 T 角点
REGISTER_LOCATOR_BUILDER(SelfLocateLT);          // L+T 组合
REGISTER_LOCATOR_BUILDER(SelfLocatePT);          // P+T 组合
REGISTER_LOCATOR_BUILDER(SelfLocate2X);          // 中圈 X 双点
```
注册后，行为树 XML 里就能直接写 `<SelfLocate mode="trust_direction"/>`（见 [模块07](../07-行为树与决策/index.md)）。

## 二、通用节点 `SelfLocate`（`locator.cpp:349`）

最常用的定位节点。它的 tick 流程：

### 1. 限频（防止每帧都定位）
```cpp
double interval = getInput<double>("msecs_interval").value();
if (msecsSince(lastSuccessfulLocalizeTime) < interval) return SUCCESS;
```
> 💡 定位是相对昂贵的计算（200 粒子 × 20 轮）。`msecs_interval` 保证两次成功定位之间至少隔一段时间，省算力。注意判据是距上次**成功**定位的时间。

### 2. 按 `mode` 设搜索范围 `constraints`
三种模式对应不同的搜索约束：

**`face_forward`**：全场搜索，朝向限定 ±45°（适合刚入场、朝向大致已知）。
```cpp
xMin/xMax = ±length/2;  yMin/yMax = ±width/2;  thetaMin/Max = ±π/4;
```

**`trust_direction`**（最常用）：信任当前位姿，只在附近小范围搜索（`locator.cpp:367`）：
```cpp
double maxDrift = msecsSince(lastLocalize)/1000 * 0.1;  // 按时间估计可能漂移量
xMin = robotPoseToField.x - maxDrift;  xMax = ... + maxDrift;  // 位置：当前±漂移
// ±5°：步态打滑会让朝向在两次定位之间漂几度
thetaMin = robotPoseToField.theta - M_PI/36;  thetaMax = ... + M_PI/36;  // 朝向：当前±5°
```
> 💡 这是"增量定位"：相信里程计提供的位姿大致对，只在它附近微调修正漂移。位置搜索范围随"距上次定位时间"扩大（漂移越久范围越大）；朝向则限定在当前值 **±5°**（`M_PI/36`）的小窗口内。

**`fall_recovery`**：摔倒爬起后，位置全场搜（`±length/2+2`、`±width/2+2`），但朝向仍信任当前值、只在 **±10°**（`M_PI/18`）内搜索（`locator.cpp:382`）：
```cpp
xMin/xMax = ±(length/2 + 2);  yMin/yMax = ±(width/2 + 2);   // 位置：全场（含边界外 2m 余量）
// ±10°：摔倒可能明显扭转身体，窗口要比 trust_direction 更宽
thetaMin = robotPoseToField.theta - M_PI/18;  thetaMax = ... + M_PI/18;  // 朝向：当前±10°
```
> 💡 摔倒后位置可能大变（整个人被撞飞/翻滚），所以位置全场搜；但爬起后朝向仍由里程计大致保持，故只在当前朝向附近搜——只是窗口比行走时的 `trust_direction` 更宽（±10° vs ±5°），以吸收摔倒带来的更大朝向扭转。

> ⚠️ **为什么朝向窗口从 ±1° 放宽到 ±5°/±10°？** 早期版本把 `trust_direction` 的朝向死死锁在 **±1°**（`M_PI/180`）。但双足行走的**步态打滑**每两次定位之间就会让朝向漂上好几度——一旦漂移超过 1°，搜索窗口里根本不含真实朝向，于是**每一次重定位都失败**，视觉再也没机会把朝向纠回来。机器人只能靠里程计"盲走"，位姿估计越漂越离谱（1v1 实测表现为**直冲球门、撞墙**）。放宽到 ±5°（行走）/±10°（摔倒后）让粒子滤波能吸收真实的朝向漂移，同时窗口仍足够窄，可以**排除镜像/对称假设**（对称场地两端相差 180°，远在窗口之外），既能纠偏又不会被歧义解带偏。

### 3. 定位 + 落地
```cpp
auto res = brain->locator->locateRobot(markers, constraints);
if (!res.success) return SUCCESS;   // 失败也返回 SUCCESS（不阻塞行为树，下次再试）
brain->calibrateOdom(res.pose.x, res.pose.y, res.pose.theta);  // 落地（见模块05·5.4）
brain->tree->setEntry<bool>("odom_calibrated", true);
brain->data->lastSuccessfulLocalizeTime = get_clock()->now();
```
> 💡 注意定位**失败也返回 `NodeStatus::SUCCESS`**。这是个重要设计——定位是"尽力而为"的辅助，失败不该让整棵行为树失败/卡住，只是这次没更新位姿、下个周期再试。

`markers` 来自 `brain->data->getMarkersForLocator()`（见 [模块05 · 5.3](../05-大脑数据与坐标系/5.3-BrainData动态数据.md)），即当前看到的角点。

## 三、入场定位 `SelfLocateEnterField`（`locator.cpp:429`）

机器人从场边入场时用。它**同时尝试"从左侧入场"和"从右侧入场"两个假设**：
```cpp
PoseBox2D cEnterLeft  = {从左边线入场的位姿范围, 朝向≈-90°};
PoseBox2D cEnterRight = {从右边线入场的位姿范围, 朝向≈+90°};
auto resLeft  = locateRobot(markers, cEnterLeft);
auto resRight = locateRobot(markers, cEnterRight);
// 谁成功用谁；都成功用残差小的
if (resLeft.success && resRight.success) res = (resLeft.residual < resRight.residual) ? resLeft : resRight;
```
> 💡 入场时机器人可能从场地任一侧进，朝向相反。与其猜，不如两个假设都算一遍，让残差更小的胜出。这是粒子滤波"多假设"思想在节点层的体现。

## 四、专用定位节点 `SelfLocate1M/2X/2T/LT/PT/Border`（`locator.cpp:498+`）

这些是针对**特定角点组合**的定位/微调节点，端口（`brain_tree.h` 或 `locator.h` 里定义）类似：
```
msecs_interval  限频
max_dist        只用近处(<max_dist)的角点，远处不可信
max_drift       定位后若和原位姿差超过 max_drift，认为定位异常，丢弃
validate        是否用其它角点交叉验证
```
典型 tick 逻辑：筛出特定类型的近距离角点 → 调 `locateRobot` → 检查结果漂移不超 `max_drift` → 落地。

- `SelfLocate1M`：单个最近角点做小幅平移微调。
- `SelfLocate2X`：用中圈的两个 X 角点（中圈位置明确，定位精度高）。
- `SelfLocate2T`/`SelfLocateLT`/`SelfLocatePT`：用 T、L、P 角点的组合。
- `SelfLocateBorder`：靠场地边界线定位。

> 💡 为什么要这么多专用节点？不同位置能看到的角点类型不同：在中圈附近能看到 X，在罚球区附近能看到 P+T，在角旗附近能看到 L。多个节点串起来"挨个尝试"，总有一个适用当前视野。`max_drift` 检查则防止某种角点组合产生的歧义解把位姿带偏。

---

## 五、`Locate` 子树：编排所有定位节点

`subtree_locate.xml`（见 [模块07](../07-行为树与决策/index.md)）把这些节点串成一个 `Locate` 子树，依次尝试：
```xml
<Sequence>
    <SelfLocate mode="trust_direction" />
    <SelfLocate1M msecs_interval="300" max_dist="3.0" max_drift="2.0" />
    <SelfLocate2T msecs_interval="300" max_dist="3.0" max_drift="2.0" />
    <SelfLocatePT .../>
    <SelfLocateLT .../>
    <SelfLocate2X .../>
    <SelfLocateBorder .../>
</Sequence>
```
> 因为每个 `SelfLocate*` 失败也返回 SUCCESS，所以 `Sequence` 会把它们**全部跑一遍**——相当于"每个周期把所有适用的定位方式都试一次，哪个成功就用哪个的结果更新位姿"。多种角点组合互补，提高定位成功率和精度。

主树里 `Locate` 子树在 SET/READY/比赛中被反复调用（前锋/守门员决策时 `decision!='find'` 才定位，避免找球时分心），详见 [模块07](../07-行为树与决策/index.md)。

---

## 小结

- 8 个 `SelfLocate*` 节点把 `locateRobot` 算法包装成可在行为树里调用的动作。
- `SelfLocate` 三模式：`face_forward`(全场、朝向±45°)、`trust_direction`(增量微调、朝向±5°)、`fall_recovery`(位置全场+朝向±10°)。朝向窗口从早期的 ±1° 放宽，是为了让视觉能纠正步态/摔倒带来的真实朝向漂移，同时仍排除镜像假设。
- 定位**失败也返回 SUCCESS**，不阻塞行为树；成功才 `calibrateOdom` 落地。
- 专用节点针对不同角点组合，`Locate` 子树把它们串起来逐个尝试，互补提精度。

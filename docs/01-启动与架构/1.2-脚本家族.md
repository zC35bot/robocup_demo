# 1.2 · 脚本家族：仿真、多机、编译

`scripts/` 目录下除了 `start.sh`/`stop.sh`，还有一族脚本，分别对应不同运行场景。理解它们就理解了项目的全部启动姿势。

| 脚本 | 场景 |
|------|------|
| `sim_start.sh` | 仿真单机器人 |
| `sim_start_multi.sh` | 仿真多机器人（6 个）|
| `start_brain.sh` / `start_vision.sh` | 单独调试大脑/视觉 |
| `build.sh` | 真机编译（带 CUDA）|
| `build_no_cuda.sh` | 无显卡编译（ONNX 推理）|
| `build_agent.sh` | 编译 agent 框架版 |

---

## 一、`sim_start.sh` —— 仿真单机器人

和 `start.sh` 几乎一样，差别全在三个 `ros2 launch` 多了 `sim:=true`：

```bash
nohup ros2 launch vision launch.py sim:=true > vision.log 2>&1 &
# nohup ros2 launch detection_converter detection_converter.launch.py > vision.log 2>&1 &
nohup ros2 launch brain launch.py sim:=true "$@"  > brain.log 2>&1 &
nohup ros2 launch game_controller launch.py sim:=true > game_controller.log 2>&1 &
```

`sim:=true` 干两件事（具体在 [1.3](./1.3-launch文件逐行.md) 展开）：
1. **启用仿真时钟** `use_sim_time=true`：节点不再用系统墙钟，而是订阅仿真器发布的 `/clock`。这样仿真暂停/加速时，所有时间戳、超时判断都跟着同步。
2. **切换到仿真用的话题/模型**：相机话题、模型路径换成仿真版（如 `sim_data_det_0126.onnx`）。

> 注意第 2 行被注释掉的 `detection_converter`：单机仿真时如果仿真器直接出标准检测消息就不需要它；多机时才显式启用（见下）。

## 二、`sim_start_multi.sh` —— 仿真多机器人（重点）

这个脚本演示了**一份代码如何驱动 6 个机器人实例**。核心是给每个实例不同的 `robot_name` 做话题命名空间隔离。

```bash
# 先启动一个共享的裁判机
nohup ros2 launch game_controller launch.py sim:=true > game_controller.log 2>&1 &

# 定义 6 个机器人：name:team_id:player_id:role
declare -a ROBOTS=(
    "robot0:29:1:goal_keeper"   # 29队 1号 守门员
    "robot1:29:2:striker"       # 29队 2号 前锋
    "robot2:29:3:striker"
    "robot3:30:1:goal_keeper"   # 30队（对手）
    "robot4:30:2:striker"
    "robot5:30:3:striker"
)
```
**两队各 3 人**：team_id 29 和 30 各一个守门员两个前锋。

```bash
for robot_config in "${ROBOTS[@]}"; do
    IFS=':' read -r robot_name team_id player_id role <<< "$robot_config"
```
**循环 + 字段拆分**：`IFS=':'` 把分隔符设为冒号，`read` 把 `"robot0:29:1:goal_keeper"` 拆成 4 个变量。

```bash
    # 每个机器人启动一个 detection_converter（仿真器输出 JSON，需转换）
    nohup ros2 launch detection_converter detection_converter.launch.py \
        input_topic:="/camera/${robot_name}_rgbd_camera/detections" \
        detections_topic:="/booster_soccer/detection/$robot_name" \
        line_segments_topic:="/booster_soccer/line_segments/$robot_name" \
        > "${robot_name}_vision.log" 2>&1 &
```
**每个机器人一个检测转换节点**：把仿真器为该机器人输出的 JSON（`/camera/robot0_rgbd_camera/detections`）转成标准消息，发到**带机器人名后缀**的话题 `/booster_soccer/detection/robot0`。详见 [模块04](../04-裁判机与通信/index.md)。

```bash
    # 每个机器人启动一个大脑
    nohup ros2 launch brain launch.py \
        sim:=true \
        team_id:=$team_id \
        player_id:=$player_id \
        role:=$role \
        robot_name:=$robot_name \
        disable_com:=false \
        "$@" > "${robot_name}_brain.log" 2>&1 &
done
```
**每个机器人一个大脑**：通过 launch 参数告诉这个大脑实例"你是 29 队 1 号守门员，名叫 robot0"。

> 💡 **多机隔离的核心机制**：大脑在 `init()` 里（见 [模块05](../05-大脑数据与坐标系/index.md)）会读 `robot_name`，把所有订阅/发布话题都加上 `/robot0` 后缀。于是 6 个大脑虽然跑同一份代码，但各自只收发自己的话题，互不串扰。视觉/检测同理。这就是 ROS2 用**命名空间**做多实例的标准玩法。
> 注意：每个机器人**没有单独启视觉节点**，而是用 `detection_converter` 接仿真器的检测——因为仿真器已经内置了"完美视觉"，不需要跑 YOLO。

## 三、`start_brain.sh` / `start_vision.sh` —— 单独调试

```bash
# start_brain.sh 核心
./scripts/stop.sh
source ./install/setup.bash
export FASTRTPS_DEFAULT_PROFILES_FILE=...
ros2 daemon stop
ros2 daemon start
ros2 launch brain launch.py sim:=true "$@"
```
只启动单个节点，且**前台运行**（没有 `nohup &`），方便直接看输出、Ctrl+C 退出。多了 `ros2 daemon stop/start`：重启 ROS2 守护进程（它缓存节点/话题发现信息），避免调试时拿到过期的发现缓存。`start_vision.sh` 类似，但注释掉了 DDS 配置那行。

## 四、编译脚本三件套

### `build.sh` —— 真机标准编译
```bash
cd `dirname $0`; cd ..
source /opt/ros/humble/setup.bash
export PYTHONWARNINGS="ignore::DeprecationWarning,ignore::UserWarning,ignore::FutureWarning"
colcon build --symlink-install --base-paths src "$@"
espeak "build complete" >/dev/null 2>&1 || echo "Build complete"
```
- `PYTHONWARNINGS=ignore...`：屏蔽编译时 Python 工具链的一堆 deprecation 噪声警告。
- `colcon build`：ROS2 多包构建（详见 [1.5](./1.5-编译与依赖.md)）。
  - `--symlink-install`：用软链接安装，改 Python/配置不用重编。
  - `--base-paths src`：只扫 `src/` 下的包。
  - `"$@"`：透传额外构建参数。
- `espeak "build complete" ... || echo`：编译完用语音播报"build complete"，没装 `espeak` 就退回打印文字（`||` 是"前者失败则执行后者"）。一个贴心的"编译完成提醒"。

### `build_no_cuda.sh` —— 无显卡编译
```bash
cd `dirname $0`
./build.sh --cmake-args -DNO_CUDA=ON -DCMAKE_BUILD_TYPE=RelWithDebInfo -DCMAKE_CXX_FLAGS="-g -rdynamic"
```
就是带特定 CMake 参数调用 `build.sh`：
- `-DNO_CUDA=ON`：定义 `NO_CUDA` 宏。视觉的检测器/分割器会据此在**编译期**选择 ONNX 后端而非 TensorRT（见 [模块03](../03-视觉模块/index.md)）。没有 N 卡也能跑。
- `-DCMAKE_BUILD_TYPE=RelWithDebInfo`：带调试信息的优化构建。
- `-DCMAKE_CXX_FLAGS="-g -rdynamic"`：`-g` 生成调试符号，`-rdynamic` 导出符号表（配合 `backward-ros` 打印崩溃栈）。

### `build_agent.sh` —— agent 框架版
```bash
cd `dirname $0`; cd ..
./booster_agent_framework/scripts/build_agent.sh ./agent --base-paths ./src
```
调用外部 `booster_agent_framework` 的构建脚本，编译 agent 模式（脱离裁判机、由 App 控制，见 [模块04](../04-裁判机与通信/index.md)）所需的版本。

# 1.5 · 编译系统 colcon 与依赖

本篇讲清楚：源码是怎么变成 `install/` 下可运行的节点的。涉及 `colcon`、`package.xml`、`CMakeLists.txt`，以及 CUDA/ONNX 编译期开关。

---

## 一、colcon：ROS2 的多包构建器

`build.sh` 的核心一行：
```bash
colcon build --symlink-install --base-paths src
```
`colcon` 会：
1. 扫描 `--base-paths src` 下每个含 `package.xml` 的目录，识别出所有 ROS2 包（brain、vision、game_controller、detection_converter、各 interface 包）。
2. 读每个包的 `<depend>` 解析**依赖关系**，拓扑排序（接口包必须先编，因为别的包要用它生成的消息类型）。
3. 按顺序对每个包跑它的构建工具（C++ 包用 `ament_cmake` → CMake → make；Python 包用 `ament_python`）。
4. 把产物装到 `install/`。

`--symlink-install` 已在 [1.2](./1.2-脚本家族.md) 讲过：用软链接，改 Python/yaml/xml 不用重编。
`--base-paths src` 限定只扫 `src/`，避免把仓库里其它地方同名的 `booster_interface` 也编进来导致冲突（根 README 特别说明了这点）。

> 💡 **依赖顺序的重要性**：`vision_interface`、`game_controller_interface`、`booster_msgs` 这些**接口包**会用 `rosidl` 把 `.msg` 编译成 C++/Python 类。brain/vision 依赖这些生成的类型，所以 colcon 必须先编接口包、再编业务包。你不用手动管顺序，colcon 按 `package.xml` 的依赖自动排。

## 二、`package.xml`：包的"身份证"与依赖清单

以 `src/brain/package.xml` 为例，关键标签：

```xml
<name>brain</name>
<buildtool_depend>ament_cmake</buildtool_depend>   <!-- 用 ament_cmake 构建 -->

<depend>rclcpp</depend>                  <!-- ROS2 C++ 客户端库 -->
<depend>behaviortree_cpp</depend>        <!-- 行为树框架（模块07核心）-->
<depend>booster_interface</depend>       <!-- 机器人底层硬件消息 -->
<depend>game_controller_interface</depend><!-- 裁判机消息 -->
<depend>vision_interface</depend>        <!-- 视觉检测消息 -->
<depend>booster_msgs</depend>            <!-- RPC 消息 -->
<depend>backward_ros</depend>            <!-- 崩溃时打印调用栈 -->
<depend>tf2 / tf2_ros / tf2_geometry_msgs</depend>  <!-- 坐标变换 -->
<depend>visualization_msgs / nav_msgs / diagnostic_msgs</depend> <!-- 可视化/栅格/标量 -->

<buildtool_depend>rosidl_default_generators</buildtool_depend>  <!-- 因为 brain 自己也定义了 Kick.msg -->
<member_of_group>rosidl_interface_packages</member_of_group>
```

`<depend>` 列出的每个依赖，对应 [模块02](../02-接口与消息/index.md) 讲的消息接口或第三方库。
> 一个细节：brain 既**依赖**别的接口包，又**自己定义**了 `msg/Kick.msg`（见 [模块02](../02-接口与消息/index.md)），所以它同时声明了 `rosidl_default_generators`——既是消息的使用者也是生产者。

`backward_ros` 值得一提：它在程序崩溃（段错误等）时自动打印**带行号的 C++ 调用栈**，配合 `build_no_cuda.sh` 里的 `-g -rdynamic` 编译选项，调试机器人这种长时间运行的程序极有用。根 README 要求 `sudo apt-get install ros-humble-backward-ros`。

## 三、`CMakeLists.txt`：brain 怎么被编出来

逐段看 `src/brain/CMakeLists.txt` 的关键部分：

```cmake
cmake_minimum_required(VERSION 3.16...3.27)
project(brain)
add_compile_options(-Wall -Wextra -Wpedantic -Wno-unknown-pragmas)  # 开警告
```

```cmake
find_package(ament_cmake REQUIRED)
find_package(rclcpp REQUIRED)
find_package(behaviortree_cpp REQUIRED)
find_package(Eigen3 REQUIRED)          # 矩阵运算（坐标变换、卡尔曼滤波）
find_package(OpenCV REQUIRED)          # 图像处理（深度图）
find_package(yaml-cpp REQUIRED)        # 读 yaml 配置
find_package(booster_interface REQUIRED)
... # 每个依赖都要 find_package
```
`find_package` 让 CMake 找到每个依赖的头文件和库。

```cmake
# 生成自定义消息 Kick 的 C++ 类型
rosidl_generate_interfaces(${PROJECT_NAME}
  "msg/Kick.msg"
  DEPENDENCIES std_msgs)             # Kick.msg 里用了 std_msgs/Header
```
这一步把 `msg/Kick.msg` 编译成 `brain::msg::Kick` C++ 类。

```cmake
# 把 src/ 下所有 .cpp 编进一个可执行文件 brain_node
file(GLOB SOURCE_FILES src/*.cpp)
add_executable(brain_node ${SOURCE_FILES})
```
`file(GLOB ...)` 抓取 `src/*.cpp` 全部源文件（brain.cpp、brain_tree.cpp、locator.cpp、…14 个），编成一个可执行 `brain_node`——就是 launch 里 `executable='brain_node'` 指的那个。

```cmake
# 让 brain_node 能找到自己生成的 Kick 消息类型
rosidl_get_typesupport_target(typesupport_target ${PROJECT_NAME} "rosidl_typesupport_cpp")
target_link_libraries(brain_node ${typesupport_target})

set_property(TARGET brain_node PROPERTY CXX_STANDARD 17)   # 用 C++17
target_link_libraries(brain_node ${OpenCV_LIBS} yaml-cpp)
target_link_libraries(brain_node ${backward_ros_LIBRARIES})
```

```cmake
# 关掉一批噪声警告（项目代码里有窄化转换、未用参数等）
set(CMAKE_CXX_FLAGS "${CMAKE_CXX_FLAGS} -Wno-narrowing -Wno-unused-parameter
    -Wno-sign-compare -Wno-missing-field-initializers -Wno-unused-variable")

# 链接所有 ROS2 依赖
ament_target_dependencies(brain_node rclcpp ... behaviortree_cpp booster_interface
    vision_interface game_controller_interface booster_msgs tf2 ... visualization_msgs)
```

```cmake
# 安装产物到 install/
install(TARGETS brain_node DESTINATION lib/${PROJECT_NAME})   # 可执行文件
install(DIRECTORY launch  DESTINATION share/${PROJECT_NAME})  # launch 文件
install(DIRECTORY config  DESTINATION share/${PROJECT_NAME})  # 配置
install(DIRECTORY behavior_trees DESTINATION share/${PROJECT_NAME})  # 行为树 XML
ament_package()
```
> 💡 注意 `install(DIRECTORY config / behavior_trees ...)`：配置文件和行为树 XML 会被**装到 `install/.../share/` 下**。配合 `--symlink-install`，它们是软链接回源码，所以你改 `config.yaml` 或 `game.xml` 不用重新 `colcon build`，重启节点即可生效。这对调参/调行为树极方便。

## 四、CUDA / ONNX 编译期开关

视觉模块的 `CMakeLists.txt`（`src/vision/`）里会检查 `NO_CUDA` 宏：
- 默认（`build.sh`）：编译 TensorRT 后端，链接 CUDA/TensorRT 库。
- `build_no_cuda.sh` 传 `-DNO_CUDA=ON`：定义 `NO_CUDA` 宏，源码里的 `#ifdef NO_CUDA` 分支选择 ONNX 后端（`detector.cc:17`），不依赖 CUDA。

```cpp
// src/vision/src/model/detector.cc
#ifdef NO_CUDA
    new YoloV8DetectorONNX(model_path);
#else
    new YoloV8DetectorTRT(model_path);
#endif
```
> 💡 这是"**编译期多态**"：不是运行时判断，而是编译时就决定只把一种后端编进去，零运行时开销。代价是换后端要重编。详见 [模块03](../03-视觉模块/index.md)。

第三方推理库 onnxruntime 需要预装（根 README 提供了 `install_onnxruntime.sh`，分 aarch64/x64 版本）。

## 五、编译产物结构

```
install/
├── setup.bash              ← start.sh 要 source 它
├── brain/
│   ├── lib/brain/brain_node          可执行文件
│   └── share/brain/
│       ├── launch/launch.py
│       ├── config/config.yaml
│       └── behavior_trees/*.xml      （软链接回源码）
├── vision/  ...
├── game_controller/ ...
└── vision_interface/  game_controller_interface/  ...  生成的消息类型
```

`source install/setup.bash` 就是把这些路径注入环境，让 `ros2 launch brain ...` 能定位到 `brain_node` 和它的 share 资源。

---

## 小结

- `colcon build` 按 `package.xml` 依赖顺序，先编接口包再编业务包。
- `package.xml` 声明依赖，`CMakeLists.txt` 描述怎么编（`file(GLOB src/*.cpp)` → `brain_node`）。
- 配置/行为树装到 share 且是软链接，改了不用重编，重启即生效。
- CUDA/ONNX 是 `-DNO_CUDA` **编译期**开关。

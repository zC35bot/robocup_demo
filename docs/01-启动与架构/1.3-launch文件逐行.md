# 1.3 · launch 文件逐行详解

`ros2 launch <包> launch.py` 执行的就是该包 `launch/launch.py`。它是 Python 写的，负责**声明参数 + 拼配置 + 启动可执行文件**。本篇逐行剖析四个 launch 文件，重点讲清楚**命令行参数 `role:=striker` 是怎么变成 C++ 里的配置的**。

---

## 一、大脑 launch：`src/brain/launch/launch.py`（最复杂，重点）

它的结构是：`generate_launch_description()` 声明一堆参数 + 一个 `OpaqueFunction`，真正的拼装逻辑在 `handle_configuration()` 里。

### 1. 为什么要用 `OpaqueFunction`？

普通 launch 在"描述阶段"参数还是未求值的占位符（`LaunchConfiguration`）。但大脑需要在启动时**读取参数的真实值去拼路径、做条件判断**（比如 role 非空才覆盖）。`OpaqueFunction` 允许注册一个回调，在 launch **真正执行时**被调用，此时能用 `context.perform_substitution()` 拿到参数的真实字符串。

```python
def generate_launch_description():
    return LaunchDescription([
        DeclareLaunchArgument('tree', default_value='game.xml', ...),
        DeclareLaunchArgument('role', default_value='', ...),
        ...
        OpaqueFunction(function=handle_configuration)  # 延迟到执行期处理
    ])
```

### 2. `DeclareLaunchArgument`：声明可命令行覆盖的参数

每一个声明，就开放一个 `xxx:=值` 的命令行入口：

```python
DeclareLaunchArgument('vision_config_path',
    default_value=os.path.join(os.path.dirname(__file__), '../../../../vision/share/vision/config'),
    description='存放 vision.yaml 的目录')
DeclareLaunchArgument('tree', default_value='game.xml',
    description='指定行为树文件名（不要带全路径）')
DeclareLaunchArgument('role', default_value='',
    description='覆盖 config.yaml 里的 player_role，如 role:=striker')
DeclareLaunchArgument('team_id', default_value='', ...)
DeclareLaunchArgument('player_id', default_value='', ...)
DeclareLaunchArgument('robot_name', default_value='', ...)
DeclareLaunchArgument('sim', default_value='false', ...)
DeclareLaunchArgument('disable_com', default_value='false', ...)
DeclareLaunchArgument('agent_mode', default_value='false', ...)
```
**默认值都是空串或 false**——意味着"不传就别覆盖配置文件"。这是关键设计：命令行只是配置文件之上的**可选覆盖层**。

### 3. `handle_configuration()` 逐段

```python
def handle_configuration(context, *args, **kwargs):
    # ① 拼出 vision 配置路径（大脑也要读相机外参 camToHead）
    vision_config_dir = context.perform_substitution(LaunchConfiguration('vision_config_path'))
    vision_config_file = os.path.join(vision_config_dir, 'vision.yaml')
    vision_config_local_file = os.path.join(vision_config_dir, 'vision_local.yaml')
```
大脑做坐标变换需要相机相对头部的外参矩阵（`camToHead`），而这个存在视觉的 `vision.yaml` 里，所以大脑也要知道它的路径。

```python
    # ② 三层大脑配置文件路径
    config_path = os.path.join(os.path.dirname(__file__), '../config')
    config_file = os.path.join(config_path, 'config.yaml')          # 基线
    config_local_file = os.path.join(config_path, 'config_local.yaml')  # 本机覆盖
    config_home_file = os.path.join(os.path.expanduser('~'), 'agents/booster_soccer/brain.yaml')  # 用户级
```

```python
    # ③ 拼行为树文件全路径
    behavior_trees_dir = os.path.join(os.path.dirname(__file__), '../behavior_trees')
    def make_tree_path(name):
        if not name.endswith('.xml'): name += '.xml'  # 没带后缀自动补
        return os.path.join(behavior_trees_dir, name)
    tree = context.perform_substitution(LaunchConfiguration('tree'))
    tree_path = make_tree_path(tree)
```
把 `tree:=game` 或 `tree:=game.xml` 都解析成 `.../behavior_trees/game.xml`。

```python
    # ④ 初始 config 字典（这些是命令行/计算出来的，优先级最高）
    config = {
        "tree_file_path": tree_path,
        "vision_config_path": vision_config_file,
        "vision_config_local_path": vision_config_local_file,
    }
```

```python
    # ⑤ 逐个处理可覆盖参数：非空才写进 config
    role = context.perform_substitution(LaunchConfiguration('role'))
    if not role == '':
        config['game.player_role'] = role        # 注意点号：嵌套 ROS 参数

    team_id = context.perform_substitution(LaunchConfiguration('team_id'))
    if not team_id == '':
        config['game.team_id'] = int(team_id)     # 转 int

    player_id = ...
    if not player_id == '': config['game.player_id'] = int(player_id)

    robot_name = ...
    if not robot_name == '': config['robot.robot_name'] = robot_name

    agent_mode = ...
    if agent_mode in ['true','True','1']: config['game.agent_mode'] = True

    sim = ...
    if sim in ['true','True','1']: config['use_sim_time'] = True   # 仿真时钟

    disableCom = ...
    if disableCom in ['true','True','1']: config['enable_com'] = False
```
> 💡 **`role:=striker` 的完整旅程**到这里就清楚了：命令行 → `start.sh` 的 `"$@"` → `ros2 launch` 的 `LaunchConfiguration('role')` → `perform_substitution` 取真实值 `"striker"` → 写进 `config['game.player_role']`。键名用点号 `game.player_role` 对应 ROS2 的嵌套参数命名（对应 `config.yaml` 里 `game:` 下的 `player_role:`）。

```python
    # ⑥ 启动 brain_node，参数按层叠顺序传入
    return [Node(
        package='brain', executable='brain_node', output='screen',
        parameters=[
            config_file,        # 第1层：config.yaml（基线默认）
            config_local_file,  # 第2层：config_local.yaml（本机覆盖）
            config_home_file,   # 第3层：~/agents/.../brain.yaml（用户覆盖）
            config              # 第4层：命令行/计算值（最高优先级）
        ]
    )]
```
**层叠的本质**：ROS2 的 `parameters` 列表按顺序加载，**后面的同名键覆盖前面的**。所以优先级从低到高是：基线 → 本机 → 用户 → 命令行。

> 💡 实战用法：把队伍通用配置放 `config.yaml`（提交 git），把"我这台 robot 腿长 1.12m、里程计标度 1.2"这种**个体差异**放 `config_local.yaml`（git 忽略），比赛临场再用命令行 `role:=...` 微调。三层互不污染。
> `output='screen'`：节点日志打到屏幕（被 `start.sh` 重定向进 `brain.log`）。

## 二、视觉 launch：`src/vision/launch/launch.py`

套路类似但更简单，主要区别：**配置文件用命令行参数传、开关用 ROS 参数传**。

```python
def handle_configuration(context, *args, **kwargs):
    default_config_path = PathJoinSubstitution([FindPackageShare('vision'), 'config']).perform(context)
    user_cfg_dir = LaunchConfiguration('vision_config_path').perform(context)
    config_path = default_config_path
    if user_cfg_dir and user_cfg_dir.strip():
        cand = user_cfg_dir.rstrip('/')
        if os.path.exists(os.path.join(cand, 'vision.yaml')):
            config_path = cand    # 用户指定目录里有 vision.yaml 就用它
        else:
            print(f"[vision launch] warning: ... fallback to {default_config_path}")
    config_file = os.path.join(config_path, 'vision.yaml')
    config_local_file = os.path.join(config_path, 'vision_local.yaml')
    config_user_file = os.path.join(os.path.expanduser('~'), 'agents/booster_soccer/vision.yaml')
```
`FindPackageShare('vision')` 找到 vision 包的安装 share 目录。允许用户用 `vision_config_path:=...` 指定外部配置目录，找不到就回退默认并打警告。

```python
    return [Node(
        package='vision', executable='vision_node', name='vision_node', output='screen',
        arguments=[config_file, config_local_file, config_user_file],  # 配置文件作为命令行参数
        parameters=[{                                                   # 开关作为 ROS 参数
            'offline_mode': offline_mode, 'show_det': show_det, 'show_seg': show_seg,
            'save_data': save_data, 'save_depth': save_depth, 'save_fps': save_fps,
            'detection_model_path': detection_model_path,
            'segmentation_model_path': segmentation_model_path,
            'use_sim_time': sim,                                        # sim:=true → 仿真时钟
            'color_topic': color_topic, 'depth_topic': depth_topic, 'intrin_topic': intrin_topic,
        }]
    )]
```
> 💡 注意视觉把**三个配置文件路径作为 `arguments`（即 `argv` 命令行参数）** 传给 `vision_node`，而 brain 是作为 `parameters`。这是两个模块作者的不同习惯——视觉的 `main.cpp` 直接 `argv[1] argv[2] argv[3]` 读三个 yaml 路径（见 [模块03](../03-视觉模块/index.md)），并自己用 `MergeYAML` 做层叠。

声明的参数包括一堆调试开关：`show_det`(弹窗显示检测)、`save_data`(存图做数据集)、`save_fps`(每秒存几帧)、`detection_model_path`(覆盖模型路径) 等，默认值如 `save_data='true'`、`save_fps='2'`。

## 三、裁判机 launch：`src/game_controller/launch/launch.py`（最简单）

```python
def generate_launch_description():
    return LaunchDescription([
        Node(package='game_controller', executable='game_controller_node', name='game_controller',
            output='screen',
            parameters=[{
                "port": 3838,                  # 裁判机广播标准端口
                "enable_ip_white_list": False, # 是否启用 IP 白名单
                "ip_white_list": ["127.0.0.1"],
            }]
        ),
    ])
```
全部硬编码、没有 `DeclareLaunchArgument`，因为裁判机节点不需要按角色/球员定制——它只是被动收 UDP 包。这也是为什么 `start.sh` 里它后面没有 `"$@"`。详见 [模块04](../04-裁判机与通信/index.md)。

## 四、检测转换 launch：`src/detection_converter/launch/detection_converter.launch.py`

仿真专用 Python 节点，三个话题名可命令行覆盖（多机时每个机器人传不同的）：

```python
input_topic_arg = DeclareLaunchArgument('input_topic',
    default_value='/camera/robot0_rgbd_camera/detections', ...)   # 仿真器输出的 JSON 话题
detections_topic_arg = DeclareLaunchArgument('detections_topic',
    default_value='/booster_soccer/detection', ...)               # 转换后输出
line_segments_topic_arg = DeclareLaunchArgument('line_segments_topic',
    default_value='/booster_soccer/line_segments', ...)

detection_converter_node = Node(
    package='detection_converter', executable='detection_converter_node.py',
    name='detection_converter', output='screen',
    parameters=[{'input_topic': ..., 'detections_topic': ..., 'line_segments_topic': ...}],
    emulate_tty=True,    # 让 Python 的彩色/行缓冲输出在日志里正常显示
)
```
`emulate_tty=True` 是给 Python 节点用的：模拟终端，让 print 的行缓冲和颜色正常工作。`sim_start_multi.sh` 就是靠覆盖这三个话题给每个机器人配独立的检测流（见 [1.2](./1.2-脚本家族.md)）。

## 小结：参数的三种传法

| 传法 | 谁用 | 怎么读 |
|------|------|--------|
| ROS `parameters`（键值字典，层叠） | brain、gc、converter | C++ `get_parameter()` |
| 命令行 `arguments`（argv） | vision | C++ `argv[i]` |
| `DeclareLaunchArgument` + `LaunchConfiguration` | 全部 | launch 内部求值后再转上面两种 |

记住**层叠优先级（基线→本机→用户→命令行）** 和 **`game.player_role` 点号对应嵌套 yaml**，就掌握了配置系统。下一篇看 C++ 进程怎么把这些参数读进去并跑起来。

# 2.2 · game_controller_interface —— 裁判机消息

比赛由一台官方"裁判机电脑"(GameController) 统一指挥：它在局域网里用 UDP 广播"现在是准备阶段还是开踢了、谁开球、几号球员被罚下"。项目里的 `game_controller` 进程把那个二进制 UDP 包翻译成 ROS2 消息，发布给大脑。本篇逐字段讲 `src/interface/game_controller_interface/msg/` 下的三个消息。

> 🏆 这三个 `.msg` 文件**严格按 RoboCup 官方 `RoboCupGameControlData.h` 的结构体逐字段对应**（每个文件首行注释都写明了这点）。字段顺序、类型、数组长度都不能改，否则 UDP 字节流就对不上了。UDP 包怎么收、怎么解析见 [模块04](../04-裁判机与通信/index.md)。

---

## 一、`GameControlData.msg` —— 完整的比赛控制包（顶层）

源码 `src/interface/game_controller_interface/msg/GameControlData.msg`（17 个字段）：

```
3  char[4] header                  # 标识符，固定 "RGme"
4  uint16 version                  # 协议版本号
5  uint8 packet_number             # 包序号（递增，溢出回绕）
6  uint8 players_per_team          # 每队人数
7  uint8 game_type                 # 比赛类型 (循环赛/淘汰赛/捡球赛)
8  uint8 state                     # ★ 主状态
9  uint8 first_half                # 1=上半场，0=否
10 uint8 kick_off_team             # 下一个开球队的队号 (或 DROPBALL)
11 uint8 secondary_state           # ★ 副状态
12 char[4] secondary_state_info    # ★ 副状态附加信息
13 uint8 drop_in_team              # 造成上次 drop-in 的队号
14 uint16 drop_in_time             # 距上次 drop-in 的秒数 (首次前为 0xffff)
15 uint16 secs_remaining           # 本半场估计剩余秒数
16 uint16 secondary_time           # 副计时（准备剩余/自由球倒计时等）
17 TeamInfo[2] teams               # 两支队伍详情
```

### 逐字段详解

| 字段 | 行 | 类型 | 含义 / 取值 | 大脑怎么用 |
|------|----|------|------|------|
| `header` | 3 | char[4] | 包头标识，固定 4 字节，用来确认"这确实是裁判机包" | 解析时校验 |
| `version` | 4 | uint16 | 协议版本，发收两端必须一致 | 兼容性校验 |
| `packet_number` | 5 | uint8 | 每发一包 +1，到 255 回绕到 0，用于丢包检测 | 判断是否漏包/乱序 |
| `players_per_team` | 6 | uint8 | 每队上场人数 | 队伍管理 |
| `game_type` | 7 | uint8 | 比赛制式：循环赛/淘汰赛/捡球赛 | 一般不影响踢球逻辑 |
| `state` | 8 | uint8 | **主状态**，见下表 | ★行为树最顶层分支 |
| `first_half` | 9 | uint8 | `1`=上半场 `0`=下半场 | 决定攻防半场朝向 |
| `kick_off_team` | 10 | uint8 | 即将开球的队号；特殊值表示 DROPBALL（争球） | `brain.cpp:1436` 算 `isKickOffSide`（本队是否开球方） |
| `secondary_state` | 11 | uint8 | **副状态**，见下表 | ★区分任意球/点球/暂停 |
| `secondary_state_info` | 12 | char[4] | 副状态的 4 字节附加信息 | 见下方专门解释 |
| `drop_in_team` | 13 | uint8 | 上次界外球/争球由哪队造成 | 罕用 |
| `drop_in_time` | 14 | uint16 | 距上次 drop-in 的秒数；首次前为 `0xffff` | 罕用 |
| `secs_remaining` | 15 | uint16 | 本半场剩余秒数（估计） | 决策时间管理 |
| `secondary_time` | 16 | uint16 | 副计时器：READY 剩余、自由球倒计时等 | 倒计时类决策 |
| `teams` | 17 | TeamInfo[2] | **定长 2** 个队伍信息（本队 + 对手），见下节 | 大脑按 `team_number` 找出"哪个是我们" |

### `state`（主状态）—— 行为的总开关

大脑 `gameControlCallback` 把 `state` 直接映射成 5 个字符串（`brain.cpp:1424-1434`）：

| `state` 值 | 名称 | 含义 | 机器人该干嘛 |
|-----------|------|------|------|
| 0 | `INITIAL` | 初始化，球员在场外等待 | 站着不动，准备入场 |
| 1 | `READY` | 准备，入场走向起始站位 | 走到开球阵型位 |
| 2 | `SET` | 就位，停下等裁判鸣哨 | 站定不动 |
| 3 | `PLAY` | 正常比赛 | 全力踢球 |
| 4 | `END` | 比赛结束 | 停止 |

> 🏆 这就是行为树最顶层的"大支"：`INITIAL/READY/SET` 阶段乱动会被判罚，只有 `PLAY` 才允许追球射门。决策细节见 [模块07](../07-行为树与决策/index.md)。映射结果写进黑板键 `gc_game_state`（`brain.cpp:1435`）。

### `secondary_state`（副状态）—— 任意球类细分

`brain.cpp:1441-1487` 用 `switch` 把它细分。原始值含义与大脑的归并策略：

| `secondary_state` 值 | 真实含义（存入 `data->realGameSubState`） | 大脑归并成 (`gameSubStateType`) | 备注 |
|-----------|------|------|------|
| 0 | `NONE` 普通比赛 | `NONE` | 正常对抗 |
| 3 | `TIMEOUT` 暂停（队伍或裁判） | `TIMEOUT` | 单独处理 |
| 4 | `DIRECT_FREEKICK` 直接任意球 | `FREE_KICK` | 置 `isDirectShoot=true`（可直接射门） |
| 5 | `INDIRECT_FREEKICK` 间接任意球 | `FREE_KICK` | 不可直接射门 |
| 6 | `PENALTY_KICK` 点球 | `FREE_KICK` | 置 `isDirectShoot=true` |
| 7 | `CORNER_KICK` 角球 | `FREE_KICK` | |
| 8 | `GOAL_KICK` 球门球 | `FREE_KICK` | 置 `isDirectShoot=true` |
| 9 | `THROW_IN` 界外球 | `FREE_KICK` | |
| 其它 | — | `FREE_KICK` | 兜底 |

> 💡 注意这里的"两层命名"设计：裁判机给的是**细分类型**（角球/界外球/点球…），但对机器人的**走位逻辑**而言，绝大多数任意球都是"球停在某处、对方要离球 X 米、我方去执行或防守"——动作模板高度一致。所以大脑把它们**统一归并成 `FREE_KICK`** 走同一套行为，只用 `realGameSubState` + `isDirectShoot` 记住细节差异（能否直接射门）。少抽象一层，行为树就简单一大截。

### `secondary_state_info`（副状态附加信息）—— 4 字节里藏两个关键量

`secondary_state_info` 是 4 字节数组，大脑只读前两个字节（`brain.cpp:1489,1492`）：

| 下标 | 含义 | 大脑用法 |
|------|------|------|
| `[0]` | 与该副状态相关的队号（如哪队执行任意球） | `isSubStateKickOffSide = (info[0] == teamId)`：判断这个任意球**是我方执行还是对方执行** → 决定上前还是退防 |
| `[1]` | 副状态的子阶段 | 映射成 `{STOP, GET_READY, SET}`（`gameSubStateMap`，`brain.cpp:1488`）：STOP=停、GET_READY=移动到攻/防位、SET=站定 |

> 💡 一个任意球不是瞬间执行的，它有"先停下 → 走到位 → 站定等哨"三个子阶段，正是 `info[1]` 在驱动。`info[0]` 则回答"这球归谁"。两个字节把"谁的球 + 进行到哪一步"说清楚了。

## 二、`TeamInfo.msg` —— 一支队伍

源码 `src/interface/game_controller_interface/msg/TeamInfo.msg`：

```
3  uint8 team_number           # 唯一队号
4  uint8 field_player_colour   # 队伍颜色
5  uint8 score                 # 本队比分
6  uint8 penalty_shot          # 点球次数计数
7  uint16 single_shots         # 各次点球成功与否的位掩码
8  uint8 coach_sequence        # 教练消息序号
9  uint8[] coach_message       # 教练消息内容 (长度 253)
10 RobotInfo coach             # 教练（也按机器人结构描述）
11 RobotInfo[] players         # 本队球员 (长度 11)
```

| 字段 | 行 | 含义 | 大脑用法 |
|------|----|------|------|
| `team_number` | 3 | 队伍唯一编号 | ★`brain.cpp:1499-1504` 用它从 `teams[2]` 里认出"哪个是我方/敌方" |
| `field_player_colour` | 4 | 球衣颜色编码 | 区分敌我视觉 |
| `score` | 5 | 当前比分 | 策略参考（如落后时更激进） |
| `penalty_shot` | 6 | 已罚点球次数 | 点球大战统计 |
| `single_shots` | 7 | 16 位的位掩码，每位代表一次点球是否罚进 | 点球大战统计 |
| `coach_sequence` | 8 | 教练消息序号（变化才表示有新消息） | 教练通信 |
| `coach_message` | 9 | 教练发给队伍的消息，长 253 字节 | 高级战术（本项目通常不用） |
| `coach` | 10 | 教练实体（复用 `RobotInfo` 结构） | 多数不用 |
| `players` | 11 | **本队所有球员**的 `RobotInfo` 数组（最多 11） | ★遍历它读每名队友的 `penalty` |

> 💡 为什么 `teams` 是定长 `[2]` 而 `players` 是变长？因为一场比赛永远是 2 队对抗（写死 2），但每队人数随级别变化（最多 11），所以用变长数组。大脑认队的逻辑：遍历 `teams[0]`/`teams[1]`，`team_number` 等于自己配置的 `team_id` 的那个就是"我方"，另一个是"敌方"（`brain.cpp:1499`）。若两个都不是自己，说明这包不关心我们，直接丢弃（`brain.cpp:1505-1509`）。

## 三、`RobotInfo.msg` —— 一名球员（`penalty` 是重点）

源码 `src/interface/game_controller_interface/msg/RobotInfo.msg`：

```
3  uint8 penalty                   # 罚则状态
4  uint8 secs_till_unpenalised     # 估计还有几秒解除处罚
5  uint8 number_of_warnings        # 警告次数
6  uint8 yellow_card_count         # 黄牌数
7  uint8 red_card_count            # 红牌数
8  bool goal_keeper                # 是否守门员
```

| 字段 | 行 | 含义 | 大脑用法 |
|------|----|------|------|
| `penalty` | 3 | **罚则状态**：`0` = 正常上场，非 0 = 被罚下（各值对应不同处罚原因，如非法移动、推人等） | ★见下 |
| `secs_till_unpenalised` | 4 | 估计还有多少秒解除处罚 | 倒计时，提前规划复位 |
| `number_of_warnings` | 5 | 累计警告次数 | 纪律统计 |
| `yellow_card_count` | 6 | 黄牌数 | 纪律统计 |
| `red_card_count` | 7 | 红牌数 | ★红牌=本场不能再上，大脑当作"替补/出局"处理 |
| `goal_keeper` | 8 | 是否守门员 | 角色判定 |

### `penalty` 对大脑意味着什么

`brain.cpp:1514-1532` 这段循环把 `penalty` 用到了极致：

```cpp
for (int i = 0; i < HL_MAX_NUM_PLAYERS; i++) {
    data->penalty[i] = static_cast<int>(myTeamInfo.players[i].penalty);
    if (myTeamInfo.players[i].red_card_count > 0)
        data->penalty[i] = PENALTY_SUBSTITUTE;           // 红牌 → 当作替补出局
    if (data->penalty[i] == PENALTY_NONE) liveCount++;   // 统计本队在场人数
    data->oppoPenalty[i] = oppoTeamInfo.players[i].penalty;
    if (oppoTeamInfo.players[i].red_card_count > 0)
        data->oppoPenalty[i] = PENALTY_SUBSTITUTE;
    if (data->oppoPenalty[i] == PENALTY_NONE) oppoLiveCount++;  // 对方在场人数
}
data->liveCount = liveCount;
data->oppoLiveCount = oppoLiveCount;
```

`penalty` 的两个核心用途：

1. 🏆 **判断自己能不能动**：`data->penalty[playerId-1]` 就是"我"的罚则状态。被罚下时机器人**必须原地不动**，乱动会延长处罚甚至吃牌——所以这是行为树的一道硬闸。
2. 🏆 **统计在场人数、动态换角色**：累计本队 `liveCount` 和对方 `oppoLiveCount`。如果守门员被罚下、场上只剩一个前锋，大脑可以临时让前锋回收去补防——人数感知是动态角色分配的输入。

> 💡 红牌特别处理成 `PENALTY_SUBSTITUTE`：普通 `penalty` 是临时的（罚几秒就回来），但红牌意味着**本场彻底出局**，必须从"在场人数"里永久扣掉，所以单独标记。`HL_MAX_NUM_PLAYERS`、`PENALTY_NONE`、`PENALTY_SUBSTITUTE` 等常量与 `data->penalty` 容器见 [模块05](../05-大脑数据与坐标系/index.md)。

## 四、本篇小结

| 关键字段 | 一句话 | 引用 |
|------|------|------|
| `state` | 比赛主状态 INITIAL/READY/SET/PLAY/END，行为树顶层分支 | `brain.cpp:1424` |
| `secondary_state` | 任意球类细分，大脑归并成 `FREE_KICK` + `isDirectShoot` | `brain.cpp:1441` |
| `secondary_state_info[0/1]` | [0]=这球归谁，[1]=子阶段(STOP/GET_READY/SET) | `brain.cpp:1489` |
| `teams[2].team_number` | 用来认出"哪个是我方" | `brain.cpp:1499` |
| `RobotInfo.penalty` | 自己能不能动 + 全队在场人数统计 | `brain.cpp:1516` |

🏆 整篇贯穿一个事实：**裁判机消息是机器人行为的"法律"**——什么时候能动、能不能直接射门、自己有没有被罚下，全由这几个字段说了算，且它们的格式被 RoboCup 官方协议钉死。

# 2.3 · booster_interface —— 机器人硬件消息

这是和机器人**身体本体**打交道的接口包：电机怎么转、IMU 测到什么姿态、脚走了多远、手柄按了哪个键、想让它走/转头/起身怎么下指令。源码在 `src/interface/booster_ros2_interface/`（编译出的包名是 `booster_interface`）。本篇按**功能分六类**讲，最后讲 `message_utils.hpp` 的工厂函数和"api_id + JSON body"的 RPC 设计理念。

> 控制链路的完整工作机制（`RobotClient` 如何封装、SDK 如何执行）见 [模块08](../08-机器人控制与底层/index.md)。本篇只讲消息契约本身。

## 一类 · 底层电机控制（下行：大脑/控制器 → 电机）

### `MotorCmd.msg` —— 单个电机的指令
源码 `src/interface/booster_ros2_interface/msg/MotorCmd.msg`：
```
1 int8 mode      # 控制模式
2 float32 q      # 目标位置（关节角，弧度）
3 float32 dq     # 目标角速度（rad/s）
4 float32 tau    # 前馈力矩（N·m）
5 float32 kp     # 位置刚度（比例增益）
6 float32 kd     # 速度阻尼（微分增益）
7 float32 weight # 该指令的混合权重
```

这是经典的**电机 PD + 前馈力矩**控制接口。实际输出力矩 ≈ `kp*(q - q_now) + kd*(dq - dq_now) + tau`：
- `q`/`dq`：希望关节到达的角度和角速度；
- `kp`/`kd`：刚度/阻尼，决定"跟得多硬"——`kp` 大则僵硬有力，小则柔顺；
- `tau`：前馈力矩，抵消重力等已知负载；
- `weight`：多指令叠加时的权重。

### `LowCmd.msg` —— 一帧的全部电机指令
源码 `src/interface/booster_ros2_interface/msg/LowCmd.msg`：
```
2 int8 CMD_TYPE_PARALLEL=0   # 常量：并联模式
3 int8 CMD_TYPE_SERIAL=1     # 常量：串联模式
7 int8 cmd_type              # 取上面两个常量之一
8 MotorCmd[] motor_cmd       # 一组电机指令
```

| 字段 | 含义 |
|------|------|
| `cmd_type` | 这批指令按**并联**(`CMD_TYPE_PARALLEL=0`)还是**串联**(`CMD_TYPE_SERIAL=1`)的电机模型解释。并联/串联对应机器人腿部不同的机械传动结构 |
| `motor_cmd` | 全身各电机的 `MotorCmd` 数组 |

> 💡 注意 `.msg` 里 `int8 NAME=value` 这种行是**常量定义**（编译后变成 C++ 的 `static const`），不是字段。这样写指令时可以用 `LowCmd::CMD_TYPE_PARALLEL` 而不是裸的魔法数字 `0`，自带语义。
>
> 💡 大脑**几乎不直接发 `LowCmd`/`MotorCmd`**。逐关节力矩控制是运动控制器（步态/平衡）的活，由 SDK 内部完成。大脑只发"语义级"指令（见下文 RPC）。`LowCmd` 在这里更多是接口的完整性和给底层控制器用。

## 二类 · 状态反馈（上行：本体 → 大脑）

### `MotorState.msg` —— 单电机反馈
源码 `src/interface/booster_ros2_interface/msg/MotorState.msg`：
```
1 int8 mode
2 float32 q          # 实际关节角（rad）
3 float32 dq         # 实际角速度（rad/s）
4 float32 ddq        # 实际角加速度（rad/s²）
5 float32 tau_est    # 估计实际力矩（N·m）
7 int8 temperature   # 电机温度，实际值，范围 -100~150（℃）
9 uint32 lost        # 丢包计数，范围 0~9999999999
12 uint32[2] reserve # [0]=错误标志(0-255), [1]=通信频率(0-800)
```

| 字段 | 含义 |
|------|------|
| `q`/`dq`/`ddq` | 实测的角度/角速度/角加速度 |
| `tau_est` | 估计的实际输出力矩 |
| `temperature` | 电机温度（℃），过热保护参考 |
| `lost` | 累计通信丢包数，反映总线健康度 |
| `reserve[0]` | 错误标志位（0-255），按位表示各类故障 |
| `reserve[1]` | 通信频率（0-800 Hz），反映该电机总线刷新率 |

> 💡 `reserve` 这种"用数组下标塞两个不同含义的量"在嵌入式协议里很常见——为了协议字段不膨胀、向后兼容，预留字段被复用，靠注释约定每个下标的含义。

### `ImuState.msg` —— 惯性测量单元
源码 `src/interface/booster_ros2_interface/msg/ImuState.msg`：
```
2 float32[3] rpy    # [0]=roll, [1]=pitch, [2]=yaw（姿态角，rad）
4 float32[3] gyro   # [0]=x, [1]=y, [2]=z（角速度，rad/s）
6 float32[3] acc    # [0]=x, [1]=y, [2]=z（加速度，m/s²）
```
机器人的"内耳平衡感"：`rpy` 给当前**姿态角**（横滚/俯仰/偏航），`gyro` 给三轴角速度，`acc` 给三轴加速度。大脑/控制器用它判断是否倾倒、维持平衡，也用 `yaw` 辅助朝向估计。

### `LowState.msg` —— 底层状态总包
源码 `src/interface/booster_ros2_interface/msg/LowState.msg`：
```
1 ImuState imu_state               # 惯性测量
2 MotorState[] motor_state_parallel # 并联模型下的全身电机状态
3 MotorState[] motor_state_serial   # 串联模型下的全身电机状态
```
一次性把 IMU + 全身电机状态打包上报。大脑订阅它（`brain.cpp:249` 话题 `/low_state`），主要取 IMU 姿态。提供并联/串联两套电机状态，对应 `LowCmd.cmd_type` 的两种模型。

### `Odometer.msg` —— 里程计
源码 `src/interface/booster_ros2_interface/msg/Odometer.msg`：
```
1 float32 x      # 累积位移 x（m，机器人自身起点坐标系）
2 float32 y      # 累积位移 y（m）
3 float32 theta  # 累积转角（rad）
```
机器人靠迈步累积估计的"我从开机到现在走了多远、转了多少角"。是大脑自定位的**重要先验**（订阅于 `brain.cpp:248` 话题 `/odometer_state`）：视觉定位会漂、会丢，里程计在两次视觉定位之间填补连续运动。其漂移特性和融合方式见 [模块06](../06-定位与球预测/index.md)。

### `FallDownState.msg` —— 摔倒状态
源码 `src/interface/booster_ros2_interface/msg/FallDownState.msg`：
```
1 uint32 IS_READY=0       # 常量：正常站立
2 uint32 IS_FALLING=1     # 常量：正在倒下
3 uint32 HAS_FALLEN=2     # 常量：已经倒地
4 uint32 IS_GETTING_UP=3  # 常量：正在爬起
6 uint32 fall_down_state       # 取上面四个常量之一
7 bool is_recovery_available   # 当前能否自主爬起
```

| 字段 | 含义 |
|------|------|
| `fall_down_state` | 四态机：`IS_READY`(站着)/`IS_FALLING`(倒下中)/`HAS_FALLEN`(趴地)/`IS_GETTING_UP`(爬起中) |
| `is_recovery_available` | 此刻是否具备自主起身条件（如电量/姿态允许） |

> 🏆 比赛里摔倒是常态。大脑要在 `HAS_FALLEN` 时立刻触发起身、在 `IS_GETTING_UP` 期间停止其它决策避免打架。注意本项目实际的摔倒/恢复状态是通过 `RawBytesMsg` 透传的（话题 `fall_down_recovery_state`，`brain.cpp:251`），`FallDownState` 是结构化的标准定义。

## 三类 · 人机交互：手柄 `RemoteControllerState`

源码 `src/interface/booster_ros2_interface/msg/RemoteControllerState.msg`，逐字段：

```
1  uint32 event   # SDL 事件类型（取自 SDL2/SDL_events.h）
2  float32 lx     # 左摇杆水平：左 -1，右 +1
3  float32 ly     # 左摇杆垂直：前 -1，后 +1
4  float32 rx     # 右摇杆水平：左 -1，右 +1
5  float32 ry     # 右摇杆垂直：前 -1，后 +1
6  bool a / b / x / y           # 四个面键
10 bool lb / rb                 # 左/右肩键(bumper)
12 bool lt / rt                 # 左/右扳机(trigger，作布尔用)
14 bool ls / rs                 # 左/右摇杆按下
16 bool back / start            # back / start 键
19 bool hat_c                   # 方向帽：居中
20 bool hat_u/d/l/r             # 方向帽：上/下/左/右
24 bool hat_lu/ld/ru/rd         # 方向帽：左上/左下/右上/右下
28 uint8 hat_pos                # 方向帽位置（取自 SDL2/SDL_joystick.h）
```

| 分组 | 字段 | 说明 |
|------|------|------|
| 事件 | `event` | SDL 事件类型枚举，标识这是按下/抬起/移动等哪类事件 |
| 摇杆 | `lx/ly/rx/ry` | 两个摇杆各两轴，归一化到 `[-1, 1]`。约定：水平左为 -1，垂直**前**为 -1 |
| 面键 | `a/b/x/y` | 四个主按钮，按下为 `true` |
| 肩键 | `lb/rb` | 左右 bumper |
| 扳机 | `lt/rt` | 左右 trigger（这里当布尔，按下/未按） |
| 摇杆按下 | `ls/rs` | 摇杆当按钮压下 |
| 功能键 | `back/start` | 菜单/开始 |
| 方向帽 | `hat_*` + `hat_pos` | 十字方向键，9 个布尔表示当前指向，`hat_pos` 是原始位编码 |

**大脑怎么订阅**（`src/brain/src/main.cpp:42`）：
```cpp
auto sub1 = node->create_subscription<booster_interface::msg::RemoteControllerState>(
    "/remote_controller_state", 10,
    bind(&Brain::joystickCallback, brain, std::placeholders::_1));
```

> 💡 摇杆轴归一化到 `[-1,1]` 是为了**和速度指令解耦**：手柄不关心机器人最大速度是多少，只表达"推到底"的比例，大脑再乘上配置的最大速度变成 `vx/vy/vyaw`。比赛中常用某个组合键（如 LT + A/B/X）在"手动遥控 / 自动决策"之间切换控制权——这部分逻辑见 [模块08](../08-机器人控制与底层/index.md)。

### `ButtonEventMsg.msg` —— 机身按键事件
源码 `src/interface/booster_ros2_interface/msg/ButtonEventMsg.msg`：
```
2  int8 PRESS_DOWN=0 / PRESS_UP=1          # 常量：按下 / 抬起
4  int8 SINGLE_CLICK=0 / DOUBLE_CLICK=1    # 常量：单击 / 双击
6  int8 TRIPLE_CLICK=0                     # 常量：三击
7  int8 LONG_PRESS_START=1                 # 常量：长按开始
8  int8 LONG_PRESS_HOLD=0                  # 常量：长按保持
9  int8 LONG_PRESS_END=1                   # 常量：长按结束
13 int8 event    # 事件类型（取上面某个常量）
14 int32 button  # 哪个按钮触发
```
机器人**机身物理按键**的事件（区别于手柄）。`event` 描述交互类型（单击/双击/长按各阶段），`button` 标识是哪个按键。

> 💡 这些常量值有重叠（`PRESS_DOWN=0` 和 `SINGLE_CLICK=0` 都是 0），说明它们**不在同一语义维度上比较**——是按不同事件类别分组的标志，消费方需结合上下文判断该按哪组常量解读。

## 五类 · 通用 API 与原始字节

### `BoosterApiReqMsg.msg` / `BoosterApiRespMsg.msg` —— 通用 API 请求/响应
源码 `BoosterApiReqMsg.msg`：
```
1 int64 api_id  # 接口编号（标识"想干什么"）
2 string body   # 参数，JSON 字符串
```
源码 `BoosterApiRespMsg.msg`：
```
1 int64 status  # 返回状态码
2 string body   # 返回内容，JSON 字符串
```

这是整套控制的**核心抽象**：不为每个动作单独定义一种消息，而是统一成"一个动作编号 `api_id` + 一段 JSON 参数 `body`"。请求 `BoosterApiReqMsg`，响应 `BoosterApiRespMsg`（`status` 是结果码，`body` 是返回数据）。

### `RawBytesMsg.msg` / `RawBytesStamped.msg` —— 原始字节透传
源码 `RawBytesMsg.msg`：
```
1 char[] msg   # 任意长度原始字节
```
源码 `RawBytesStamped.msg`：
```
1 std_msgs/Header header  # 时间戳
2 char[] msg              # 原始字节
```
"逃生通道"：当某些数据没有专门的结构化消息时，直接塞一段字节流透传。带 `Stamped` 的多了时间戳。本项目的摔倒恢复状态就走 `RawBytesMsg`（`brain.cpp:251` 订阅 `fall_down_recovery_state`）。

### 服务 `.srv`
ROS2 的 `.srv` 是**请求-响应**式（不像话题是单向广播），用 `---` 分隔请求与响应两段。

`RpcService.srv`：
```
1 booster_interface/BoosterApiReqMsg msg   # 请求：一个 API 请求
2 ---
3 booster_interface/BoosterApiRespMsg msg  # 响应：一个 API 响应
```

`AgentService.srv`：
```
1 string body   # 请求：一段字符串
2 ---
3 string body   # 响应：一段字符串
```
`RpcService` 是同步 RPC（发请求、等响应）；`AgentService` 是 agent 模式下的纯字符串进出通道。

> 💡 话题(`.msg`) vs 服务(`.srv`) 的取舍：控制指令在比赛里要**高频、低延迟、发了就走**（fire-and-forget），所以本项目实际走**话题**（见 [2.4](./2.4-booster_msgs与Kick.md) 的 `RpcReqMsg`），而不是会阻塞等待的 `.srv` 服务。`.srv` 留作需要确切回执的场景。

## 六类 · `message_utils.hpp` —— 指令构造工厂

源码 `src/interface/booster_ros2_interface/include/booster_interface/booster_interface/message_utils.hpp`。它提供一组**工厂函数**，把"我想让机器人做某事"打包成一个 `BoosterApiReqMsg`。

### 地基：`ConstructMsg`
`message_utils.hpp:14`：
```cpp
msg::BoosterApiReqMsg ConstructMsg(LocoApiId api_id, std::string json_body) {
    msg::BoosterApiReqMsg msg;
    msg.api_id = static_cast<int64_t>(api_id);  // 动作编号
    msg.body = json_body;                       // JSON 参数
    return msg;
}
```
所有工厂函数最终都收敛到这里：填 `api_id`、填 `body`。`LocoApiId` 是 SDK 定义的运动控制接口枚举。

### 模板版：`CreateMsg`（推荐用法）
`message_utils.hpp:21-29`：
```cpp
template <LocoApiId api_id, typename ParameterType = void, typename... Args>
msg::BoosterApiReqMsg CreateMsg(Args &&...args) {
    if constexpr (std::is_same_v<ParameterType, void>) {
        return ConstructMsg(api_id, "");                          // 无参动作
    } else {
        ParameterType param(std::forward<Args>(args)...);
        return ConstructMsg(api_id, param.ToJson().dump());       // 参数对象自动序列化为 JSON
    }
}
```
编译期把 `api_id` 当模板参数固定下来，类型安全；有参数时用参数对象的 `ToJson().dump()` 自动序列化，免去手写 JSON。

### 便捷封装（被标记为遗留但仍在用）
`message_utils.hpp:35-95` 一组语义化函数，每个对应一个动作：

| 工厂函数 | 行 | 对应 `LocoApiId` | 干什么 |
|------|----|------|------|
| `CreateChangeModeMsg(mode)` | 35 | `kChangeMode` | 切换机器人模式 |
| `CreateMoveMsg(vx, vy, vyaw)` | 40 | `kMove` | 以速度走路（前进/横移/转向） |
| `CreateRotateHeadMsg(pitch, yaw)` | 45 | `kRotateHead` | 转头到指定俯仰/偏航角 |
| `CreateRotateHeadWithDirectionMsg(pd, yd)` | 50 | `kRotateHeadWithDirection` | 按方向增量转头 |
| `CreateWaveHandMsg(idx, action)` | 55 | `kWaveHand` | 挥手 |
| `CreateLieDownMsg()` | 60 | `kLieDown` | 趴下（无参） |
| `CreateGetUpMsg()` | 63 | `kGetUp` | 起身（无参） |
| `CreateMoveHandEndEffector(...)` | 66/75 | `kMoveHandEndEffector` | 移动手末端到目标位姿 |
| `CreateControlGripperMsg(...)` | 83 | `kControlGripper` | 控制夹爪 |
| `CreateSwitchHandEndEffectorControlModeMsg(on)` | 91 | `kSwitchHandEndEffectorControlMode` | 切换手末端控制模式 |

### 💡 "api_id + JSON body" 的 RPC 设计理念

这套设计是整个控制层的灵魂，值得专门说透：

1. **决策层和运动层彻底解耦**：大脑只说"以 vx 速度前进"（一个语义指令），不关心要怎么协调几十个关节的力矩去实现行走。把"语义指令 → 关节力矩"的脏活全留给 SDK 的运动控制器。
2. **加新动作零成本**：想支持一个新动作（比如"鞠躬"）？只要约定一个新的 `api_id`、定义它的 JSON 参数格式即可，**消息定义 `BoosterApiReqMsg` 一个字都不用改**，发收两端也不用重新生成代码。对比之下，如果每个动作一种消息类型，每加一个动作全链路都要改、重编。
3. **跨语言/跨进程友好**：`api_id` 是整数、`body` 是字符串 JSON，任何语言都能轻松构造和解析，天然适合分布式 RPC。
4. **代价**：失去了编译期字段检查——JSON body 写错字段名编译器不会报错，只能运行时发现。模板版 `CreateMsg` 用"参数对象 + `ToJson()`"把这个风险收回了一部分。

> `RobotClient` 如何把这些 `BoosterApiReqMsg` 进一步装进 `RpcReqMsg` 信封发出去，见 [2.4](./2.4-booster_msgs与Kick.md)；端到端控制流程见 [模块08](../08-机器人控制与底层/index.md)。

## 本篇小结

| 类别 | 关键消息 | 方向 |
|------|------|------|
| 底层控制 | `LowCmd` / `MotorCmd`（PD+前馈力矩） | 下行（多由 SDK 用） |
| 状态反馈 | `LowState` / `MotorState` / `ImuState` / `Odometer` / `FallDownState` | 上行 → 大脑 |
| 人机交互 | `RemoteControllerState`（摇杆归一化到 ±1）/ `ButtonEventMsg` | 上行 → 大脑 |
| 手部 | `HandParam` ⊂ `HandCommand` ⊂ `HandDdsMsg` | 下行 |
| 通用 API | `BoosterApiReqMsg`(api_id+JSON) / `BoosterApiRespMsg` / `RawBytes*` / `.srv` | 双向 |
| 工厂 | `message_utils.hpp`：`ConstructMsg` / `CreateMsg<>` / `CreateMoveMsg` 等 | 构造下行指令 |

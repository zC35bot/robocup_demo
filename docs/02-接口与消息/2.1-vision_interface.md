# 2.1 · vision_interface —— 视觉的输出消息

视觉进程看完一帧图像、推完神经网络后，把"我看见了什么、在哪"打包成 `vision_interface` 包里的消息发出来，大脑订阅这些话题。本篇把 `src/interface/vision_interface/msg/` 下 **10 个 `.msg` 文件逐字段**讲透，重点剖析 `DetectedObject` 的**三套位置坐标**。

> 这些消息怎么算出来（神经网络、地面投影几何、深度对齐）属于视觉模块的内部，见 [模块03](../03-视觉模块/index.md)。本篇只讲"契约"——字段的含义、单位、谁用。

---

## 一、`DetectedObject.msg` —— 一个被检测到的物体（最核心）

源码 `src/interface/vision_interface/msg/DetectedObject.msg`（共 13 个字段）：

```
1  string label
2  string color
3  float64 confidence
4  int64 xmin
5  int64 ymin
6  int64 xmax
7  int64 ymax
8  float32[] target_uv
9  float32[] received_pos
10 float32[] position
11 float32[] position_projection
12 float32[] position_cam
13 int32 position_confidence
```

逐字段：

| 字段 | 行 | 类型 | 含义 / 单位 / 取值 | 谁用 |
|------|----|------|------|------|
| `label` | 1 | string | 类别标签：`"Ball"`(球)、`"Goalpost"`(球门柱)、`"Person"`/`"Opponent"`(人/对手)、`"LCross"`/`"TCross"`/`"XCross"`(场地线交叉点) 等 | 大脑按 label 分流到不同处理逻辑 |
| `color` | 2 | string | 颜色/附加描述（如球衣颜色），多数物体留空 | 区分敌我等 |
| `confidence` | 3 | float64 | 检测置信度，约 `0~1`（部分模型按 0~100），越大越可信 | 大脑做置信度过滤（见球后处理流水线） |
| `xmin`/`ymin`/`xmax`/`ymax` | 4-7 | int64 | 物体在图像里的**像素包围框**(bounding box) 左上/右下角，单位=像素 | 大脑存进 `GameObject.boundingBox`（`brain.cpp:1979`） |
| `target_uv` | 8 | float32[] | 用于定位的**精确像素点** `[u, v]`（不一定是框中心，比如球用最低接地点）。长度为 2 时才有效 | `brain.cpp:1973` 取 `target_uv[0/1]` 存进 `precisePixelPoint` |
| `received_pos` | 9 | float32[] | 拍这帧时机器人/头部的位姿快照，便于把"当时看到的"坐标变换回去 | 时序对齐 |
| `position` | 10 | float32[] | **位置·综合估计** `[x, y, z]`，视觉融合后给出的"最佳猜测" | 备用，本项目大脑当前未直接读 |
| `position_projection` | 11 | float32[] | **位置·地面投影** `[x, y]`（假设物体贴地 z=0，用相机几何把像素点投到地面平面） | ★大脑实际使用的距离来源（`brain.cpp:1986-1987`） |
| `position_cam` | 12 | float32[] | **位置·相机深度** `[x, y, z]`（用深度相机/点云直接测得，相机坐标系） | 备用，深度可靠时可校正 |
| `position_confidence` | 13 | int32 | 位置估计的置信等级（整数分档） | 位置可信度判断 |

### 💡 为什么要三套位置？（本篇最重要的一段）

同一个物体，视觉给出三种不同来源的坐标，因为**没有任何单一方法在所有距离都准**：

| 坐标 | 怎么来的 | 优点 | 缺点 |
|------|----------|------|------|
| `position_projection`（地面投影） | 把物体的接地像素点，按相机外参/俯仰角，几何投影到 z=0 的地面平面 | 对**贴地物体**（球、对手脚、场地标记）几何精确，**远处也不失效** | 物体不贴地（如悬空、人的躯干）会偏；依赖头部俯仰角准确 |
| `position_cam`（相机深度） | 深度相机/双目点云直接读这个像素的深度 | 不依赖"贴地"假设，近处准 | 远处深度噪声大甚至无数据；反光/黑色物体测不准 |
| `position`（综合） | 视觉内部对前两者做融合/择优 | "一个值省心" | 融合策略是黑盒，大脑想自己控制时不够灵活 |

> 💡 **本项目大脑的选择**：在 `getGameObjects()` 里有一行注释明明白白——`brain.cpp:1985` "Do not use depth measurement, directly use projection distance"，紧接着 `gObj.posToRobot.x = obj.position_projection[0]`。也就是说**大脑只信地面投影**。原因：足球场里大脑关心的几乎都是贴地物体（球、对手、线），地面投影在全场范围都几何精确，而深度相机在 3 米外就开始飘。把 `position_cam`/`position` 留在消息里，是给"未来需要深度时"留的扩展口。
>
> 拿到 `position_projection` 后，大脑顺手算出极坐标：`range = norm(x,y)`、`yawToRobot = atan2(y,x)`，再用机器人自身位姿 `transCoord` 变换到**场地坐标系** `posToField`（`brain.cpp:1990-1999`）。位置坐标系的来龙去脉见 [模块05](../05-大脑数据与坐标系/index.md)。

🏆 比赛相关：球的接地点定位精度直接决定能否对准球门踢。地面投影对"球"这个完美贴地的球体最友好，这也是大脑敢于"只用投影"的底气。

## 二、`Detections.msg` —— 一帧的全部检测

源码 `src/interface/vision_interface/msg/Detections.msg`：

```
1  std_msgs/Header header
2  DetectedObject[] detected_objects
3  float32[] radar_x
4  float32[] radar_y
5  float32[] corner_pos
```

| 字段 | 行 | 含义 | 谁用 |
|------|----|------|------|
| `header` | 1 | 标准 ROS2 头：`stamp`(时间戳) + `frame_id`(坐标系名)。时间戳用于把检测和当时的机器人位姿对齐 | `brain.cpp:1962` `timePointFromHeader(detections.header)` |
| `detected_objects` | 2 | 这一帧识别到的**所有物体数组**，每个是上面的 `DetectedObject` | 大脑遍历它，逐个转成 `GameObject` |
| `radar_x` / `radar_y` | 3-4 | 雷达式扫描数据：把视野按角度切片，给出每个方向最近障碍距离，用于避障的快速参考 | 障碍/避障逻辑 |
| `corner_pos` | 5 | 图像**四个角投影到地面**的坐标，描述"当前视野在场地上覆盖的范围"（梯形四顶点） | 判断目标是否在视野内 |

**大脑怎么订阅**（`src/brain/src/brain.cpp:246`）：

```cpp
detectionsSubscription = create_subscription<vision_interface::msg::Detections>(
    "/booster_soccer/detection" + topic_suffix, SUB_STATE_QUEUE_SIZE,
    bind(&Brain::detectionsCallback, this, _1));
```

- 话题名 `/booster_soccer/detection`，多机器人时拼上 `topic_suffix`（如 `/robot0`），实现一份代码多实例隔离。
- 回调 `detectionsCallback` 内部调用 `getGameObjects()` 把消息翻译成大脑内部的 `GameObject` 列表。

## 三、`LineSegments.msg` —— 场地白线

源码 `src/interface/vision_interface/msg/LineSegments.msg`：

```
1  std_msgs/Header header
2  float32[] coordinates     #
3  float32[] coordinates_uv  #
```

| 字段 | 行 | 含义 |
|------|----|------|
| `header` | 1 | 时间戳 + 坐标系 |
| `coordinates` | 2 | 线段端点的**三维/地面坐标**，扁平存放 `[x0,y0,x1,y1, x0,y0,x1,y1, ...]`，每 4 个数一条线段 |
| `coordinates_uv` | 3 | 同样的线段在**图像里的像素坐标** `[u0,v0,u1,v1, ...]`，便于回投可视化 |

**大脑怎么订阅**（`brain.cpp:247`）：

```cpp
subFieldLine = create_subscription<vision_interface::msg::LineSegments>(
    "/booster_soccer/line_segments" + topic_suffix, SUB_STATE_QUEUE_SIZE,
    bind(&Brain::fieldLineCallback, this, _1));
```

> 💡 为什么场地线如此重要？足球场地除了中圈和球门，**绝大多数地方长得一模一样**，机器人光靠"看到草地"没法知道自己站在哪。但白线交叉点（L/T/X 角）是有限且位置已知的"地标"。大脑把看到的线段和已知的场地线地图做匹配，反推出"我现在站在场地的哪个坐标、朝哪"——这就是**自定位**。算法见 [模块06](../06-定位与球预测/index.md)。

## 四、其余 7 个消息（速查）

### `Ball.msg` —— 精简版球位置
源码 `Ball.msg`：
```
1 std_msgs/Header header
2 float64 x
3 float64 y
4 float64 confidence
```
只关心球的平面坐标 `(x, y)` 和置信度，是 `DetectedObject` 的轻量替代，某些只需要球的链路用它。

### `CalParam.msg` —— 相机标定补偿
源码 `CalParam.msg`：
```
1 float32 pitch_compensation # degs   俯仰角补偿，单位：度
2 float32 yaw_compensation   # degs   偏航角补偿，单位：度
3 float32 z_compensation     # m      高度补偿，单位：米
```
在线微调相机外参用——机器人头部装配存在机械误差，这三个补偿值修正俯仰/偏航/高度，让地面投影更准。

> 💡 这直接影响 `position_projection` 的精度：俯仰角差 1 度，远处球的投影位置就能偏几十厘米。`pitch_compensation` 就是用来"拧准"这个角度的旋钮。

### `DetectionsTrack.msg` —— 带跟踪的检测
源码 `DetectionsTrack.msg`，比 `Detections` 多一个 `ball_type`：
```
1 std_msgs/Header header
2 string ball_type  # "yolo" or "pysot" or "none"
3 DetectedObject[] detected_objects
4 float32[] radar_x
5 float32[] radar_y
6 float32[] corner_pos
```
`ball_type` 标明这一帧的球是**检测器**(yolo，逐帧识别)还是**跟踪器**(pysot，跟住上一帧的球)给的，还是没有(`none`)。跟踪器在球被部分遮挡时仍能维持目标。

### `SegmentationResult / SegmentationLine / SegmentationPoint` —— 语义分割
三个配套消息，描述像素级分割出来的线和点：

`SegmentationResult.msg`：
```
1 std_msgs/Header header
2 SegmentationLine[] lines
3 SegmentationPoint[] points
```

`SegmentationLine.msg`（一条分割线，同时带场地坐标与像素坐标）：
```
1 string label              线类别
2 float64 x1, y1, x2, y2    两端点的场地坐标
6 float64 u1, v1, u2, v2    两端点的像素坐标
```

`SegmentationPoint.msg`（一个分割点）：
```
1 string label   点类别
2 float64 x, y    场地坐标
4 float64 u, v    像素坐标
```

> 每个几何元素都"双坐标"（场地 `x/y` + 像素 `u/v`），就是为了既能拿来定位（用场地坐标）又能叠回原图调试（用像素坐标）。

### `ImageHeadPosition.msg` —— 图像 + 头部位姿
源码 `ImageHeadPosition.msg`：
```
1 std_msgs/Header header
2 geometry_msgs/Pose position   拍这帧时头部/相机的位姿
3 sensor_msgs/Image image       原始图像
```
把"图像"和"拍它时头在哪"绑在一起，主要用于离线采数据、标定、回放。

## 五、本篇小结

| 你应该记住的 | 一句话 |
|------|------|
| 三套位置 | `position`(综合) / `position_projection`(地面投影) / `position_cam`(深度)；**大脑只用投影**（`brain.cpp:1986`） |
| 为什么用投影 | 足球关心的都是贴地物体，地面投影全场几何精确、远处不失效 |
| `Detections` | 一帧全部物体 + 雷达 + 视野四角，订阅于 `/booster_soccer/detection` |
| `LineSegments` | 白线段（场地坐标 + 像素坐标），喂给自定位 |
| 双坐标惯例 | 几乎所有几何字段都同时给场地坐标和像素坐标，定位 + 可视化两用 |

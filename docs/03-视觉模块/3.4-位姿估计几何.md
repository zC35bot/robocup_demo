# 3.4 · 位姿估计与几何（核心）精读

这是视觉模块**最核心、最巧妙**的一篇。神经网络只告诉你"图像里第 320 行第 200 列有个球"，但决策需要"球在场地上 (x=2.1, y=-0.5)"。本篇讲清楚这步是怎么做到的：相机内参 `Intrinsics` 的投影/反投影、坐标变换链、`CalculatePositionByIntersection` 射线-地面求交逐行、为何用 z=0 假设、四种专用估计器，以及场地线的三维 RANSAC 拟合。

涉及源码：`base/intrin.cpp`/`.h`、`base/pose.cpp`/`.h`、`pose_estimator/pose_estimator.cpp`/`.h`、`pose_estimator/hungarian_matching.hpp`。

---

## 一、相机内参 `Intrinsics`

`base/intrin.h`。描述"三维点 ↔ 像素"的映射，含焦距 `fx,fy`、主点 `cx,cy`、畸变系数、畸变模型（三选一）：

```cpp
enum DistortionModel {
    kNone = 0,                 // 无畸变（仿真 / 已矫正图像）
    kBrownConrady = 1,         // OpenCV 标准畸变（正向）
    kInverseBrownConrady = 2   // 反向 Brown-Conrady（RealSense 等）
};
```

### Project：三维 → 像素（intrin.cpp:43）

无畸变 / kInverseBrownConrady 走最简单的针孔模型：
```cpp
uv.x = fx * point.x / point.z + cx;
uv.y = fy * point.y / point.z + cy;
```
kBrownConrady 则先算径向 `r²,r⁴,r⁶` 和切向畸变项，把归一化坐标 `x'/y'` 扭曲后再乘焦距（`intrin.cpp:52-65`）。

### BackProject：像素 → 相机系射线（intrin.cpp:70）

这是反过程，给一个像素求它对应的"从相机出发的射线方向"（深度未知时 `depth=1`）：

```cpp
case kNone:                        // 直接反针孔
    xyz.x = (u - cx) * depth / fx;
    xyz.y = (v - cy) * depth / fy;
    xyz.z = depth;
case kBrownConrady:                // 用 cv::undistortPoints 去畸变
    cv::undistortPoints(...);
case kInverseBrownConrady:         // 牛顿迭代反解畸变（10 次）
    for (i < 10) { ...icdist... x = (xo - delta_x) * icdist; }
```

> 💡 为什么 `kInverseBrownConrady` 要**迭代 10 次**？畸变是个非线性正向映射，求逆没有闭式解。代码用不动点迭代：先假设无畸变，反复用畸变公式修正归一化坐标，10 次经验上足够收敛（`intrin.cpp:95` 注释 "10 iterations determined empirically"）。

> 💡 `Project` 和 `BackProject` 是一对：检测/位姿估计用 `BackProject`（像素→射线），场地线把 3D 端点投回图像用 `Project`（验证可视化）。无畸变情况下两者互逆。

内参可静态来自 `vision.yaml`（`Intrinsics(YAML::Node)` 构造，`intrin.cpp:9`），也可由 `CameraInfo` 话题动态更新（见 [3.1](./3.1-节点与主流水线.md)）。

## 二、`Pose`：4×4 齐次变换

`base/pose.h`。内部就是一个 `cv::Mat 4×4`（`CV_32F`），表示旋转 + 平移。提供多种构造（欧拉角、四元数、旋转矩阵、ROS TF 消息）和运算符：

| 运算 | 含义 |
|------|------|
| `pose1 * pose2` | 位姿串联（矩阵乘） |
| `pose * point3f` | 把三维点从一个坐标系变到另一个（齐次乘） |
| `pose.inverse()` | 逆变换 |
| `getRotationMatrix()` / `getTranslationVec()` | 取旋转 / 平移 |

欧拉角构造（`pose.cpp:7`）按 `R = Rz·Ry·Rx`（yaw·pitch·roll）合成旋转。`getEulerAnglesVec`（`pose.cpp:97`）反解 RPY，含奇异点（万向锁）处理。

## 三、坐标变换链（关键）

主流水线每帧算（`vision_node.cpp:345`）：

```cpp
Pose p_eye2base = p_head2base * p_headprime2head_ * p_eye2head_;
```

读法是**从右往左**串联——一个相机系下的点 `p_eye`，要变到 base 系：

```
p_base = p_head2base · p_headprime2head · p_eye2head · p_eye
          └───────┘    └──────────────┘   └─────────┘
          头→机器人      在线补偿(标定误差)    相机→头
          实时随头转动    pitch/yaw/z 微调     固定外参(标定)
```

| 变换 | 来源 | 何时变 |
|------|------|--------|
| `p_eye2head_` | 标定得到的固定外参，存 `vision.yaml` 的 `extrin` 4×4 | 启动加载，标定时才改 |
| `p_headprime2head_` | 在线补偿，`Pose(0,0,z_comp,0,pitch,yaw)` | 由 `CalParam` 消息实时更新 |
| `p_head2base` | 头相对身体的位姿，来自 `/head_pose` | 每帧实时（头一直在转） |

> 💡 为什么要单独拆出 `p_headprime2head_` 这一层"补偿"？相机外参标定不可能 100% 准，机器人头部装配也有公差。把"剩余的俯仰/偏航/高度偏差"独立成一个可在线调的小变换，比赛现场只需微调这三个数（通过 `/booster_soccer/cal_param` 话题，`CalParamCallback`），不必重新标定整套外参、不必重启。`field` 上发现球的落点系统性偏前/偏后，就调 pitch_compensation。

坐标系约定（`vision.yaml` 注释）：相机系 `x右 y下 z前`，头部系 `x前 y左 z上`。这套链路与 [模块05 · 坐标系](../05-大脑数据与坐标系/index.md) 衔接。

## 四、`CalculatePositionByIntersection` 射线-地面求交（逐行）

`pose_estimator.cpp:8`。**全模块最核心的 20 行**。

```cpp
cv::Point3f CalculatePositionByIntersection(const Pose &p_eye2base, const cv::Point2f target_uv, const Intrinsics &intr) {
    cv::Point3f normalized_point3d = intr.BackProject(target_uv);     // ① 像素 → 相机系射线方向

    cv::Mat mat_obj_ray = (cv::Mat_<float>(3,1) << normalized_point3d.x, .y, .z);
    cv::Mat mat_rot   = p_eye2base.getRotationMatrix();               // ② R: 相机→base 的旋转
    cv::Mat mat_trans = p_eye2base.toCVMat().col(3).rowRange(0,3);    //    t: 相机原点在 base 系的位置

    cv::Mat mat_rot_obj_ray = mat_rot * mat_obj_ray;                  // ③ 把射线方向转到 base 系

    float denom = mat_rot_obj_ray.at<float>(2,0);                     // ④ 分母 = 射线方向的 z 分量
    if (std::abs(denom) < 1e-6f) return cv::Point3f(0, 0, 0);         //    近水平射线 → 不与地面相交，返回 0
    float scale = -mat_trans.at<float>(2,0) / denom;                  //    求 z=0 的比例

    cv::Mat mat_position = mat_trans + scale * mat_rot_obj_ray;       // ⑤ 交点
    return cv::Point3f(...);
}
```

### 逐步几何含义

- **①** `BackProject(uv)` 得到相机系下一条从光心出发、穿过该像素的射线方向 `d_eye`（`z=1` 归一化）。
- **②** `p_eye2base` 拆成旋转 `R` 和平移 `t`。`t` 正是**相机光心在 base 系的坐标**。
- **③** `d_base = R · d_eye`，把射线方向旋到 base 系。
- **④** 射线参数方程：`P(s) = t + s · d_base`。要它落在地面 `P.z = 0`：
  ```
  t.z + s · d_base.z = 0   ⟹   s = -t.z / d_base.z
  ```
  这里的分母 `d_base.z`（即 `mat_rot_obj_ray.at<float>(2,0)`）是射线方向在 base 系的竖直分量。
- **⑤** 代回得交点 `P = t + s · d_base`，即物体在 base 系的三维位置（理论上 `P.z≈0`）。

> 💡 **除零护栏**（`pose_estimator.cpp:17-18`，`393253a` 加固）：当射线几乎水平（`|d_base.z| < 1e-6`）时，视线与地面平行、根本不相交，`s = -t.z / d_base.z` 会除以≈0 得到 `±inf`/`NaN`，进而污染后续所有位置计算，甚至把无穷值发进检测消息让大脑决策失控。加固后一旦分母过小就直接返回 `(0,0,0)`。这种情况在图像上边缘（看向远方地平线附近）的像素做地面求交时真实存在，必须挡住。

### 为什么用 z=0 假设？

> 💡 对**贴地物体**（球、对手脚、场地角点、罚球点）来说，它们的真实高度就是 0（在地面上）。所以"这条视线打到地面的哪一点"就是物体的真实位置——**几何精确，零额外假设**。而且它**完全不依赖深度相机**：深度在 3~4 米外噪声大、甚至无返回，而地面求交在整个视野范围内一致可靠。这就是本系统位置估计稳健的根本原因。

> 🏆 球是贴地滚的、对手是站在地上的、场地线画在地上——RoboCup 的关键目标几乎全在地面 `z=0` 上。z=0 假设几乎覆盖了所有需要定位的目标，深度只在近距离需要消歧义时作为"精化"补充（见下文 EstimateByDepth）。

## 五、四种专用估计器

基类 `PoseEstimator`（`pose_estimator.h:14`）提供 `EstimateByColor`（地面求交，默认取框中心）和 `EstimateByDepth`（默认返回空）。三个子类按物体几何各自重写。`vision_node.cpp` 按类别名 `get_estimator` 挑选（[3.1](./3.1-节点与主流水线.md)）。

| 估计器 | 用于 | EstimateByColor 取点 | EstimateByDepth |
|--------|------|---------------------|-----------------|
| `PoseEstimator`(默认) | 兜底 | **框中心** | 空 |
| `BallPoseEstimator` | Ball | **框底边中点** | 点云球面 RANSAC 拟合球心 |
| `HumanLikePoseEstimator` | Person/Opponent/Goalpost | **框底边中点** | 点云平面拟合后求交 |
| `FieldMarkerPoseEstimator` | LCross/TCross/XCross/PenaltyPoint | 框中心（可选霍夫精化） | — |

### 1. 默认 `PoseEstimator`（pose_estimator.cpp:23）

```cpp
cv::Point2f target_uv = cv::Point2f(bbox.x + bbox.width/2, bbox.y + bbox.height/2);  // 框中心
cv::Point3f target_xyz = CalculatePositionByIntersection(p_eye2base, target_uv, intr_);
```

### 2. `BallPoseEstimator`（pose_estimator.cpp:47）

颜色估计取**框底边中点**：
```cpp
cv::Point2f target_uv = cv::Point2f(bbox.x + bbox.width/2, bbox.y + bbox.height);  // 注意是 +height，底边
```

> 💡 为什么球取框**底边**而不是中心？球是个立体的圆，检测框框住的是它的轮廓。球**接触地面的点**在框的底部正中，那才是 z=0 上的真实落点。取框中心会把位置算得偏远（中心比接触点高半个球径）。

深度估计 `EstimateByDepth`（`pose_estimator.cpp:55`）做球面 RANSAC：
```cpp
auto pose = EstimateByColor(...);                          // 先用颜色估个位置
if (norm(pose) > filter_distance_) return pose;            // 太远(>1.2m)就不用深度，深度不准
CreatePointCloud(cloud, depth, rgb, bbox, intr_);          // 框内建点云
DownSamplePointCloud(...); ClusterPointCloud(...);         // 下采样 + 聚类
for (cluster : clusters) {
    SphereFitting(sphere, confidence, cluster, ..., radius_);  // RANSAC 拟合球面
    if (confidence > 0.5) {
        pose = p_eye2base * Pose(sphere center);           // 球心 → base 系
        if (check_ball_height_ && pose.z < -0.03) continue;  // 球心陷地下，是误检
        return pose;
    }
}
return Pose();                                              // 深度拟合失败 → 空（会被丢弃）
```

> 💡 球面拟合是用深度反过滤"假球"的利器：真球的点云能拟合出一个半径≈0.109m 的球面（`radius_`），背景误检拟不出来。`vision_node.cpp:421` 里若 `use_depth_ && Ball && 深度估计为空` 就丢弃该检测。点云细节见 [3.5](./3.5-点云与图像桥.md)。

### 3. `HumanLikePoseEstimator`（pose_estimator.cpp:102）

颜色估计同样取框底（脚/柱底接触地面）。深度估计（`pose_estimator.cpp:109`）改用**平面拟合**：
```cpp
if (pose.x > 3) return pose;                               // 远处(>3m)不用深度
CreatePointCloud → DownSample → PointCloudNoiseRemoval(去噪) → PlaneFitting(平面 RANSAC);
if (plane_coeffs.size() < 4) return pose;                  // ① 平面拟合失败 → 退回颜色估计
// 用拟合出的平面与"框中心射线"求交
float denominator = ray.x*a + ray.y*b + c;
if (std::abs(denominator) < 1e-6f) return pose;            // ② 射线与平面平行 → 退回颜色估计
float scale = -d / denominator;                            // 射线-平面求交（一般平面，非 z=0）
pose = p_eye2base * Pose(ray * scale);
```

> 💡 人/门柱是高大物体，正面是一个近似竖直平面。拟合这个平面再与视线求交，比单纯地面求交更能定位"物体表面"位置，对近距离的高物体更稳。

> 💡 这里 `393253a` 补了**两道护栏**（`pose_estimator.cpp:133`、`144-146`）：
> 1. **平面系数校验**：原来只有一句 `// TODO(GW): add plane fitting res check`。若点云太差、`PlaneFitting` 没拟合出有效平面（`plane_coeffs` 不足 4 个 `[a,b,c,d]`），后面 `plane_coeffs[0..3]` 会**越界读**。现在 `size() < 4` 时直接退回已有的颜色估计 `pose`。
> 2. **除零护栏**：分母 `ray.x*a + ray.y*b + c` 是视线方向与平面法向的点积，为 0 表示视线与平面平行、无交点，`-d / 0` 会得 `inf`/`NaN`。过小就退回颜色估计。两道护栏都**优雅降级**回地面求交结果，而非返回垃圾位置。

### 4. `FieldMarkerPoseEstimator`（pose_estimator.cpp:441）

角点默认取框中心地面求交。若 `refine: true`，调 `RefineFieldMarkerCenterDetection`（`pose_estimator.cpp:249`）做**霍夫直线精化**：

```mermaid
flowchart LR
    A["裁剪角点框周围区域"] --> B["Canny 边缘"]
    B --> C["HoughLinesP 检测线段"]
    C --> D["按到中心距离过滤线段"]
    D --> E["合并共线线段<br/>CanBeMerged + FitLineRANSAC"]
    E --> F["求两两线段交点"]
    F --> G["取靠近中心(<45px)的交点平均"]
    G --> H["作为精化后的角点像素"]
```

> 💡 为什么角点要精化？L/T/X 角点是两条场地线的交叉，检测框的几何中心不一定正好是线的交点（框可能偏）。用霍夫找出那几条线、求它们的交点，能把角点像素定位得更准——而角点是自定位的关键地标，像素差一点、地面求交后场地坐标就差不少。`scale_factor = 2.0 / distance` 让远处角点用更松的合并阈值（`pose_estimator.cpp:445`）。

`CanBeMerged`（`pose_estimator.cpp:162`）判两线段是否共线：角度差 < 5°（`CV_PI/36`）且点到线距离和小于阈值。`get_colliner_idx`（`pose_estimator.cpp:238`）递归把传递共线的线段归成一组。

## 六、场地线三维拟合 `FitFieldLineSegments`

`pose_estimator.cpp:495`。把分割出的线条轮廓投到地面、拟合成三维线段，供大脑自定位。

```cpp
for (const auto &contour : contours) {
    seg.area = cv::contourArea(contour);
    for (point : contour)
        seg.contour_3d_points.push_back(CalculatePositionByIntersection(p_eye2base, point, intr));  // ① 逐点地面求交
}
for (auto &seg : field_line_segments) {
    if (seg.area < area_threshold) continue;               // ② 面积太小丢弃
    FitLineRANSAC(seg.line_model, seg.inlier_count, ..., seg.contour_3d_points, 0.07, 100);  // ③ 3D RANSAC 直线
    // ④ 算平均点，沿拟合方向 dp 排序，取首尾作端点
    std::sort(contour_3d_points, [&](a,b){ return dp.dot(a-avg) < dp.dot(b-avg); });
    seg.end_points_3d = {首, 尾};
    seg.end_points_2d = { intr.Project(p_base2eye * 首), intr.Project(p_base2eye * 尾) };  // 投回像素验证
}
```

### 三维 RANSAC `FitLineRANSAC`（pose_estimator.cpp:452）

```cpp
for (k < max_trails) {                    // 100 次随机试验
    随机取两点 p1,p2，方向 dp = (p2-p1) 归一化;
    for (每个点) {
        float d = |v.y*dp.x - v.x*dp.y|;  // 点到 (p1,dp) 直线的距离（XY 平面）
        if (d < dist_threshold) inlier_count++;  // 阈值 0.07m
    }
    if (inlier_count > best) 记录最佳直线;
}
```

> 💡 为什么在**三维**做 RANSAC 而不是图像 2D？场地线投到地面后是真实的直线段，在 3D 拟合得到的是物理意义明确的"场地坐标系下的线段端点"，可以直接和已知的场地画线（[3.6 field.yaml](./3.6-标定与配置.md)）匹配做自定位。注意距离只用 XY 分量（`v.y*dp.x - v.x*dp.y`）——因为线在地面上、z 本就≈0。

回到 `ProcessSegmentationData`（[3.1](./3.1-节点与主流水线.md)），内点率 < 0.25 的线段被丢弃，其余端点的 3D + 2D 坐标打包进 `LineSegments` 消息。

## 七、匈牙利匹配 `hungarian_matching.hpp`（地标关联工具）

`pose_estimator/hungarian_matching.hpp`。提供把"观测到的角点"与"已知场地地标"做最优一一匹配的工具（munkres 算法）。

`MarkerCoordinates` 的距离 = **欧氏距离 + 名字距离 + 邻居拓扑距离**（`hungarian_matching.hpp:69`）：
- `EuclideanDistance`：两地标位置差。
- `NameDistance`：名字相含（如都是 LCross）算 0.5。
- `NeighborDistance`：比较各自最近 2 个邻居的拓扑结构。

`HungarianMatching`（`hungarian_matching.hpp:75`）建代价矩阵、用 `munkres_cpp::Munkres` 求最小代价匹配，代价 < 0.5 的配对才采纳。

> 💡 把欧氏距离、类型、邻居拓扑三者加权，是为了在"多个相似角点"里靠**周围结构**消歧义——光看位置容易把相邻的 L 角点配错，加上"它周围还有哪些角点、是什么类型"就能区分。这是给自定位/标定做数据关联的辅助工具。

## 小结

- `Intrinsics`：`Project`（3D→像素）、`BackProject`（像素→射线，畸变下迭代/去畸变）。
- 坐标链 `p_eye2base = p_head2base · p_headprime2head · p_eye2head`：固定外参 + 在线补偿 + 实时头位姿。
- **射线-地面求交**是核心：像素→射线→转 base 系→与 `z=0` 求交。对贴地物几何精确、不靠深度。
- 四种估计器：球取框底、人/门柱取框底+平面拟合、角点取中心+霍夫精化、默认取中心。深度仅做近距精化与假球过滤。
- 场地线：轮廓逐点投地面 → 三维 RANSAC 直线 → 端点 → 喂自定位。

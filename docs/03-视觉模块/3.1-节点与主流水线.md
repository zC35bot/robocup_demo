## 一、入口 `main.cpp` 逐行

```cpp
int main(int argc, char **argv) {
    if (argc < 2) { ... return -1; }                 // main.cpp:4
    rclcpp::init(argc, argv);                          // main.cpp:8
    rclcpp::NodeOptions options;
    options.use_intra_process_comms(true);             // main.cpp:13
    auto node = std::make_shared<booster_vision::VisionNode>(node_name, options);
    std::string config_template_path = argv[1];        // main.cpp:17
    std::string config_path = (argc > 2) ? argv[2] : "";
    node->Init(config_template_path, config_path);     // main.cpp:22
    rclcpp::executors::MultiThreadedExecutor executor(rclcpp::ExecutorOptions(), 4);  // main.cpp:24
    executor.add_node(node);
    executor.spin();
}
```

**第 4 行**：要求至少一个参数——配置模板路径。launch 实际传了三个（见下文）。

**第 13 行**：`use_intra_process_comms(true)` 开启**进程内零拷贝通信**。

> 💡 为什么要零拷贝？一帧 RGB 图几 MB，深度图也不小。如果走默认的序列化/反序列化，每帧都要拷贝两次。进程内通信让同一进程内的发布/订阅直接传共享指针，省掉拷贝。这对图像这类大消息收益极大。

**第 24 行**：`MultiThreadedExecutor(..., 4)` 用 **4 个线程**跑回调。配合 `Init` 里创建的 4 个互斥回调组（`MutuallyExclusive`），让"收彩色图做检测""收彩色图做分割""收深度""收位姿"这几类回调能**并行**，而同类回调内部串行（不会自己踩自己）。

> 💡 回调组的设计：彩色图既要喂检测又要喂分割，这两件事都很重（各自要跑一次神经网络推理）。把它们放进不同回调组（`sub_opt_1` vs `sub_opt_2`），多线程执行器就能让检测和分割**真正并行**，而不是排队。

## 三、`ProcessData` 一帧检测的完整步骤

`ColorCallback` 收到彩色图后，先经 `data_syncer_->getSyncedDataBlock` 凑齐深度和位姿，再调 `ProcessData(synced_data, detection_msg)`（`vision_node.cpp:322`）。逐步骤：

### 步骤 4：逐物体位姿估计（vision_node.cpp:410-451）

`get_estimator(class_name)`（`vision_node.cpp:354`）按类别名挑估计器：
- `Ball` → `ball`
- `Person`/`Opponent`/`Goalpost` → `human_like`
- 含 `Cross` 或 `PenaltyPoint` → `field_marker`
- 其余 → `default`

对每个检测：
```cpp
Pose pose_obj_by_color = pose_estimator->EstimateByColor(p_eye2base, detection, color);  // 地面求交
Pose pose_obj_by_depth = pose_estimator->EstimateByDepth(p_eye2base, detection, color, depth_float); // 深度精化
detection_obj.position_projection = pose_obj_by_color.getTranslationVec();  // 地面投影位置
detection_obj.position            = pose_obj_by_depth.getTranslationVec();  // 深度位置
```

> 💡 每个物体输出**两套位置**：`position_projection`（地面求交，永远有值、贴地物精确）和 `position`（深度估计，可能为空）。大脑可按物体类型选用。球若开了深度且深度估计失败（返回空 `Pose()`），直接丢弃这个检测（`vision_node.cpp:421`）——这是用深度反过滤"假球"。

还会填入检测框像素坐标、置信度（×100 转百分制）、`received_pos`（当时的头位姿，6 维）。若是 `Opponent` 且配了颜色分类器，裁剪框区域分类队服颜色塞进 `detection_obj.color`。裁剪前先把检测框与图像边界求交做**越界钳制**（`vision_node.cpp:449-451`）：

```cpp
cv::Rect safe_bbox = detection.bbox & cv::Rect(0, 0, color.cols, color.rows);
if (safe_bbox.empty()) continue;                 // 与图像无交叠 → 跳过该检测
cv::Mat crop = color(safe_bbox);
```

> 💡 为什么要钳制？检测框可能贴着甚至越出图像边缘（尤其对手站在画面边上时），直接用原始 `bbox` 去 `color(bbox)` 取子图会因超出 `Mat` 范围而抛异常、让整条流水线崩溃。用 `bbox & 图像矩形`把框裁回合法范围，交集为空（框完全在图外）时直接 `continue` 跳过，避免空裁剪。

### 步骤 5：四角 + 中心地面投影（vision_node.cpp:454-461）


> 💡 把图像四角投影到地面，等于告诉大脑"这一帧相机看到了场地上的哪块梯形区域"。这对自定位、判断"哪里没看到"很有用。

#### 标注检测帧的显示与发布（vision_node.cpp:494-530）

只要 `show_det_` 或 `pub_det_image_` 有一个开启，就先把彩色图转 RGB、调 `YoloV8Detector::DrawDetection` 画出检测框得到 `img_out`，之后本地弹窗与远程发布**共用同一张标注帧**：

```cpp
if (show_det_ || (pub_det_image_ && detection_img_pub_)) {
    cv::Mat color_rgb;
    cv::cvtColor(color, color_rgb, cv::COLOR_BGR2RGB);
    cv::Mat img_out = YoloV8Detector::DrawDetection(color_rgb, detections_for_display);

    if (show_det_) {                         // 本地调试：需要接显示器
        cv::imshow("Detection", img_out);
        cv::waitKey(1);
    }

    // 远程发布：把标注帧编成 JPEG 走 CompressedImage
    if (pub_det_image_ && detection_img_pub_) {
        cv::Mat img_bgr;
        cv::cvtColor(img_out, img_bgr, cv::COLOR_RGB2BGR);          // img_out 是 RGB，imencode 要 BGR
        std::vector<uchar> buf;
        std::vector<int> params = {cv::IMWRITE_JPEG_QUALITY, 80};   // JPEG 质量 80
        if (cv::imencode(".jpg", img_bgr, buf, params)) {
            sensor_msgs::msg::CompressedImage cmsg;
            cmsg.header = detection_msg.header;                     // 复用检测消息的 header（时间戳/坐标系一致）
            cmsg.format = "jpeg";
            cmsg.data = std::move(buf);
            detection_img_pub_->publish(cmsg);
        }
    }
}
```

> 💡 这段是 `038c595` 补齐的功能。此前 `detection_img_pub_` 只是**声明了但从未接线**，`show_det_` 也只能把标注帧渲染到 `cv::imshow` 本地窗口——要求机器人上接着显示器，比赛/远程调试时根本看不到。现在把同一张标注帧编码成 JPEG，以 `CompressedImage` 发到 `/booster_soccer/detection_image`，远程查看器（如通过 WebSocket 桥接的 Foxglove）就能直接渲染出带检测框的画面。

> 💡 几个实现细节及其原因：
> 1. **发 JPEG 而非原始帧**：机器人相机的原始 `/boostercamera/head/rgb` 是 nv12（YUV）编码，Foxglove 根本无法渲染；而 JPEG 是通用可解码格式。质量取 80 是画质与带宽的折中——标注帧要经 2.4GHz WiFi 传出，压缩能大幅降码率。
> 2. **RGB→BGR 再编码**：`img_out` 是 RGB（`DrawDetection` 在 RGB 上画的），而 `cv::imencode` 按 OpenCV 惯例期望 BGR，不转的话发出去的图会红蓝互换。
> 3. **默认关闭 + 条件发布**：整段用 `pub_det_image_`（默认 false）门控，比赛里不编码、不发布，零额外开销；`imencode` 失败时也不会发出损坏消息。
> 4. **复用 `detection_msg.header`**：让标注帧与检测结果共享同一时间戳和坐标系，远程端能把框和图对齐。

## 四、`ProcessSegmentationData` 分割流水线

与检测平行的另一条线（`vision_node.cpp:582`），只需彩色 + 位姿：

```cpp
Pose p_eye2base = p_head2base * p_headprime2head_ * p_eye2head_;
auto segmentations = segmentor_->Inference(color);            // YOLOv8-seg
for (auto &seg : segmentations) {
    auto line_segs = FitFieldLineSegments(p_eye2base, intr_, seg.contour, line_segment_area_threshold_);
    for (auto line_seg : line_segs) {
        float inlier_pct = (float)line_seg.inlier_count / line_seg.contour_2d_points.size();
        if (inlier_pct < 0.25) continue;                     // RANSAC 内点率太低，丢弃
        // 把端点的 3D 场地坐标 + 2D 像素坐标塞进消息
    }
}
field_line_pub_->publish(field_line_segs_msg);
```

`FitFieldLineSegments` 的三维 RANSAC 拟合见 [3.4](./3.4-位姿估计几何.md)。内点率 < 0.25 的线段（轮廓点不够"成一条直线"）被丢弃。

## 六、相机内参动态更新 `CameraInfoCallback`

`vision_node.cpp:792`。订阅 `CameraInfo` 话题，把 `intr_` 从静态 yaml 值改为相机实时上报值。

```cpp
float fx = msg->k[0]; float fy = msg->k[4]; float cx = msg->k[2]; float cy = msg->k[5];
// 畸变模型判定（vision_node.cpp:808-829）：
//   d 为空 → kNone
//   d 全约 0 → kNone
//   distortion_model=="plumb_bob" → kInverseBrownConrady, 否则 kBrownConrady
bool should_update = first_update ||
    std::abs(intr_.fx - fx) > 0.1 || ... ;   // 内参变化超过 0.1 像素才更新
if (should_update) {
    intr_ = new_intr;
    pose_estimator_map_.clear();             // 用新内参重建所有估计器！
    pose_estimator_map_["default"]      = std::make_shared<PoseEstimator>(intr_);
    pose_estimator_map_["ball"]         = std::make_shared<BallPoseEstimator>(intr_); ...
}
static int received_count = 0;
if (++received_count >= 5) { camera_info_sub_.reset(); }  // 稳定 5 次后退订
```

> 💡 三个设计点：
> 1. **重建估计器**：内参 `intr_` 是按值拷进每个估计器的，改内参必须重建它们，否则估计器还用着旧内参。这里复用了 `config_node_`（步骤 1 存的）。
> 2. **0.1 像素阈值**：避免内参在小数位抖动时反复重建。
> 3. **5 次后退订**：内参是相机出厂固定值，启动时确认几次就够了，之后退订省 CPU/省回调线程占用。

## 七、launch 关键点

`launch.py` 用 `OpaqueFunction` 在运行时拼配置路径（`launch.py:8`）：默认包内 `config/`，可被 `vision_config_path` 覆盖。声明的命令行参数（`show_det`/`show_seg`/`pub_det_image`/`save_data`/`sim`/`color_topic` 等）都透传成节点参数。`save_data` 默认 `true`、`sim`/`pub_det_image` 默认 `false`。launch 总体机制见 [模块01 · 1.3](../01-启动与架构/1.3-launch文件逐行.md)。

## 小结

- `main` 做三件事：开零拷贝、4 线程执行器、调 `Init`。
- `Init` 顺序：合并配置 → 读开关 → 相机参数 → 检测/分割器 → 同步器/日志 → 位姿估计器 → 订阅发布。
- `ProcessData` 一帧：时间差检查 → 坐标链 → 推理 → 后处理 → 逐物体两套位姿 → 四角投影 → 发布。
- 检测与分割是**两条独立流水线**，各有同步器、各在不同回调组并行。
- 内参可由 `CameraInfo` 动态更新并重建估计器，稳定后退订。

# 3.2 · 数据同步 DataSyncer 精读

视觉模块的输入来自**三个不同步的话题**：彩色图、深度图、头部位姿。它们各自异步到达、时间戳互不相同。要算出"物体在场上哪里"，必须先把"同一时刻"的彩色 + 深度 + 位姿凑到一起。这件事由 `DataSyncer` 完成。

涉及源码：`src/vision/include/booster_vision/base/data_syncer.hpp`、`src/vision/src/base/data_syncer.cpp`。

---

## 一、为什么需要同步？

机器人头在转、身体在走，相机的"眼睛到场地"的变换 `p_eye2base` 时刻在变。如果用 t=100ms 的彩色图配 t=140ms 的头位姿去做射线-地面求交（[3.4](./3.4-位姿估计几何.md)），头在这 40ms 里转动的角度会让结果偏出几十厘米。所以：

> 🏆 比赛中机器人头部扫视很快（找球时左右甩头）。位姿和图像哪怕差 30~40ms，落到 3~4 米外的球上就是十几厘米误差，足以让踢球踢偏。时间同步是位置精度的前提。

## 二、数据结构

### `DataBlock<T>` —— 带时间戳的数据（data_syncer.hpp:13）

```cpp
template <typename T>
struct DataBlock { T data; double timestamp; };

using ColorDataBlock = DataBlock<cv::Mat>;
using DepthDataBlock = DataBlock<cv::Mat>;
using PoseDataBlock  = DataBlock<Pose>;
```

三种数据各裹一个时间戳。

### `SyncedDataBlock` —— 对齐后的三元组（data_syncer.hpp:30）

```cpp
struct SyncedDataBlock {
    ColorDataBlock color_data;
    DepthDataBlock depth_data;
    PoseDataBlock  pose_data;
};
```

它的拷贝构造（`data_syncer.hpp:32`）对 `cv::Mat` 做了 `copyTo` **深拷贝**——避免多线程下底层像素被别处改动。

### `DataBuffer<MaxLength, T>` —— 定长环形缓冲（data_syncer.hpp:54）

```cpp
template <int MaxLength, typename T, typename Allocator = std::allocator<T>>
class DataBuffer : public std::deque<T, Allocator> {
public:
    DataBuffer() : std::deque<T, Allocator>(MaxLength) {}   // 预填 MaxLength 个空元素
    void push_back(const T &value) {
        if (this->size() == MaxLength) this->pop_front();    // 满了就丢最旧
        std::deque<T, Allocator>::push_back(value);
    }
};
```

继承 `std::deque`，重写 `push_back`：**满了就从头部弹出最旧的**——这就是环形缓冲。注意构造时**预填了 `MaxLength` 个默认元素**，所以缓冲一开始就是"满"的（里面是空 `Mat`/默认 `Pose`，时间戳 0）。

```cpp
const int kDepthBufferLength = 30;    // 深度缓冲：30 帧
const int kPoseBufferLength  = 500;   // 位姿缓冲：500 个
```

> 💡 缓冲大小为何这么选？深度图和彩色图大致同频（~30FPS），所以深度缓冲 30 帧≈1 秒历史足够。位姿话题发布频率高得多（头位姿可能上百 Hz），要在同样的时间窗口内覆盖到，就需要 500 这种更大的缓冲。

## 三、写入：`AddDepth` / `AddPose`

```cpp
void DataSyncer::AddDepth(const DepthDataBlock &depth_data) {   // data_syncer.cpp:35
    if (!enable_depth_) return;                                  // 没开深度直接丢弃
    std::lock_guard<std::mutex> lock(depth_buffer_mutex_);
    depth_buffer_.push_back(depth_data);
}
void DataSyncer::AddPose(const PoseDataBlock &pose_data) {       // data_syncer.cpp:41
    std::lock_guard<std::mutex> lock(pose_buffer_mutex_);
    pose_buffer_.push_back(pose_data);
}
```

各有独立互斥锁。由 `DepthCallback`/`PoseCallBack` 在各自回调线程调用——它们只入缓冲，不触发处理。

## 四、核心：`getSyncedDataBlock` 最近匹配（逐行）

彩色图到达时调用（`data_syncer.cpp:94`）。这是本篇的重头戏。

```cpp
SyncedDataBlock DataSyncer::getSyncedDataBlock(const ColorDataBlock &color_data) {
    SyncedDataBlock synced_data;
    synced_data.color_data = color_data;                  // 彩色就是基准帧

    // ① 把两个缓冲整体拷一份出来（持锁时间最短）
    DepthBuffer depth_buffer_cp;
    { std::lock_guard<std::mutex> lock(depth_buffer_mutex_); depth_buffer_cp = depth_buffer_; }
    PoseBuffer pose_buffer_cp;
    { std::lock_guard<std::mutex> lock(pose_buffer_mutex_);  pose_buffer_cp = pose_buffer_; }

    double color_timestamp = color_data.timestamp;

    // ② 深度：倒序找时间戳最接近的
    if (enable_depth_) {
        double smallest_depth_timestamp_diff = DBL_MAX;
        auto depth_end = depth_buffer_cp.rbegin() + std::min(static_cast<size_t>(kDepthBufferLength), depth_buffer_cp.size());
        for (auto it = depth_buffer_cp.rbegin(); it != depth_end; ++it) {
            double diff = std::abs(it->timestamp - color_timestamp);
            if (diff < smallest_depth_timestamp_diff) {
                smallest_depth_timestamp_diff = diff;
                synced_data.depth_data = *it;
                synced_data.depth_data.timestamp = it->timestamp;
                it->data.copyTo(synced_data.depth_data.data);   // 深拷贝
            } else {
                break;                                           // 时间差开始增大 → 提前退出
            }
        }
    }

    // ③ 位姿：同样倒序找最接近的
    double smallest_pose_timestamp_diff = DBL_MAX;
    auto pose_end = pose_buffer_cp.rbegin() + std::min(static_cast<size_t>(kPoseBufferLength), pose_buffer_cp.size());
    for (auto it = pose_buffer_cp.rbegin(); it != pose_end; ++it) {
        double diff = std::abs(it->timestamp - color_timestamp);
        if (diff < smallest_pose_timestamp_diff) {
            smallest_pose_timestamp_diff = diff;
            synced_data.pose_data = *it;
        } else {
            break;
        }
    }
    return synced_data;
}
```

### 拆解 ① 先整体拷贝再处理

持锁只做一次 `depth_buffer_cp = depth_buffer_`（deque 整体拷贝），随后释放锁，再慢慢遍历副本。

> 💡 为什么不直接持锁遍历？因为遍历里还有 `copyTo` 深拷贝大图像，耗时较长。持锁久了会卡住 `AddDepth`/`AddPose`（它们在别的回调线程等锁），拖慢收图。"先拷出来再处理"把临界区压到最短，是经典的锁优化。

### 拆解 ② / ③ 倒序遍历 + 提前退出

`rbegin()` 是缓冲**最新**的一端，往前是越来越旧。遍历时：
- 时间差 `diff` 一路减小 → 越来越接近彩色帧的时刻 → 更新最佳匹配。
- 一旦 `diff` 开始**增大** → 说明已经越过了最接近点 → `break` 立即退出。

> 💡 为什么能"一旦变大就退出"？因为缓冲里的时间戳是**单调递增**入队的，从新到旧遍历，`|t - color_t|` 是一个先减后增的 V 形（彩色帧时刻通常落在某两个缓冲帧之间）。从最新端往旧走，差值先减到谷底再单调增。所以见到第一次增大就能确定谷底已过，不必遍历完整个缓冲。这把平均复杂度从 O(N) 降到"只看最新几个"。

> 💡 为什么从**最新端**开始而不是最旧端？因为彩色帧刚到，它对应的位姿/深度大概率也是缓冲里**较新**的几个。从新端找，通常走两三步就命中谷底退出。

### 边界细节：遍历右边界钳制到缓冲实际长度

遍历的右边界不再是裸的 `rbegin() + kBufferLength`，而是：

```cpp
auto depth_end = depth_buffer_cp.rbegin() + std::min(static_cast<size_t>(kDepthBufferLength), depth_buffer_cp.size());
auto pose_end  = pose_buffer_cp.rbegin()  + std::min(static_cast<size_t>(kPoseBufferLength),  pose_buffer_cp.size());
```

> 💡 为什么加这层 `std::min`？原来的写法 `rbegin() + kBufferLength` 默认"缓冲长度恒等于常量 `kBufferLength`"。这个假设在缓冲被预填成定长时通常成立，但一旦缓冲的实际大小小于常量（例如缓冲尚未填满、被清空重建、或常量与 `DataBuffer` 模板长度不一致），`rbegin() + kBufferLength` 就会**越过 `rend()`**——迭代器指向缓冲外的非法内存，随后 `it->timestamp` 解引用即为未定义行为，轻则读到垃圾时间戳选错帧、重则崩溃。用 `std::min(kBufferLength, size())` 把右边界钳到"常量"和"实际长度"里较小的那个，保证遍历永远不会跨出缓冲。这是 `393253a` 安全加固的一部分。

其余细节：

- 深度做了 `copyTo` 深拷贝，位姿是值类型（`Pose` 内含 `cv::Mat`，`*it` 拷贝即可）直接赋值。
- 若 `enable_depth_` 为 false（分割流水线的 `seg_data_syncer_`、或检测不开深度），跳过深度匹配，`depth_data` 留空。

## 五、离线版 `getSyncedDataBlock()`（无参）

`data_syncer.cpp:46`。离线回放时不靠话题，而是从磁盘按时间戳读取数据集：

```cpp
SyncedDataBlock DataSyncer::getSyncedDataBlock() {
    if (time_stamp_list_.empty()) return SyncedDataBlock();      // 无数据集 → 直接返回空块
    std::string color_file = data_dir_ + "/color_" + ts + ".jpg";
    std::string depth_file = data_dir_ + "/depth_" + ts + ".png";
    std::string pose_file  = data_dir_ + "/pose_"  + ts + ".yaml";
    // 分别 imread / imread(ANYDEPTH) / YAML::LoadFile
    data_index_ = (data_index_ + 1) % time_stamp_list_.size();   // 循环回放
}
```

开头的 `if (time_stamp_list_.empty()) return SyncedDataBlock();`（`data_syncer.cpp:47`，`393253a` 加固）是一道空集合护栏：若 `LoadData` 没扫到任何数据文件，`time_stamp_list_` 为空，后面 `time_stamp_list_[data_index_]` 会越界访问、`% time_stamp_list_.size()` 会除零。提前返回空块避免崩溃。

配套的 `LoadData`（`data_syncer.cpp:9`）用正则 `color_([0-9]+\.[0-9]+)\.jpg` 扫目录解析时间戳并排序。这套数据集正是 [3.6 数据记录](./3.6-标定与配置.md) 里 `DataLogger` 存下来的，文件名格式一一对应。

## 六、与主流水线的衔接

| 触发 | 调用 | 效果 |
|------|------|------|
| 彩色到达 `ColorCallback` | `data_syncer_->getSyncedDataBlock(color)` | 凑齐深度+位姿 → `ProcessData` |
| 彩色到达 `SegmentationCallback` | `seg_data_syncer_->getSyncedDataBlock(color)` | 凑齐位姿 → `ProcessSegmentationData` |
| 深度到达 `DepthCallback` | `data_syncer_->AddDepth` | 仅入缓冲 |
| 位姿到达 `PoseCallBack` | 两个 syncer 都 `AddPose` | 仅入缓冲 |

凑齐后，`ProcessData` 会检查时间差，超过 40ms 打警告（`vision_node.cpp:327`），提示同步质量。

---

## 小结

- `DataSyncer` = 定长环形缓冲（深度 30 / 位姿 500）+ 时间戳最近匹配。
- 以**彩色帧为节拍**，深度/位姿只入缓冲、不触发处理。
- 匹配算法：**先整体拷贝缩短临界区**，再**从最新端倒序遍历**，利用时间戳单调性"一旦差值变大就提前退出"。
- 缓冲大小按各话题频率定：位姿频率高所以缓冲更大。
- 离线模式从磁盘按时间戳读，与 `DataLogger` 存盘格式对应。
